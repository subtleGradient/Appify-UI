humane-web-dream.web / compatibility and native web APIs

Native Web Compatibility

.web bundles should be ordinary client-side web sites: HTML, CSS, JavaScript, forms, workers, storage, and service workers. Web.app may provide local hosting and safe filesystem affordances, but compatibility should come from web-native APIs that degrade naturally in a normal browser.

Decision

  • Make stable webspace origins the foundation for persistent native browser state.
  • Support real service workers first-class; do not emulate service workers with window globals.
  • Support NavigateEvent.formData where WebKit exposes the Navigation API.
  • Use normal Request, Response, FormData, URLSearchParams, and location objects for form and request compatibility.
  • Remove preferred dependence on window.AppifyHost.request and window.__WEB_APP_REQUEST__ from examples and future APIs.
  • Never execute bundle-provided server code. Dynamic-looking routes are static shadow files or client-side service worker responses.

Era Targets

Era Conventions to Preserve Native Mechanism
Web 1.0 Handwritten HTML, tables, frames-ish navigation, image maps, plain forms, fossil CGI-looking action URLs. Static files, normal form submission, URLSearchParams for GET-compatible state, and explicit .cgi.html shadow pages.
Web 1.5 Early AJAX, XMLHttpRequest, small HTML fragments, progressive enhancement, no build step. Same-origin fetch/XHR, fragment HTML files, History API, and bundle-owned client scripts.
Web 2.0 / PWA Client routing, offline behavior, background request handling, rich local state, workers. Service worker fetch events, CacheStorage, IndexedDB, OPFS, cookies, Web Workers, and persistent webspace origin state.

POST Compatibility Model

  • The preferred request interception API is a real service worker fetch event with event.request and event.respondWith().
  • For forms that can safely become GET-compatible state, the service worker should answer POST with a redirect to the static landing page plus query parameters.
  • The landing page can synchronously read new URLSearchParams(location.search) during initial parsing, including from old document.write code.
  • For non-GET-compatible bodies, the service worker should synthesize a Response or fetch a static shadow page and inject bundle-owned inert data such as <script type="application/json" id="posted-request">.
  • Bundle code that handles modern navigation should feature-detect navigation and use NavigateEvent.formData for same-document form flows.
  • Web.app should keep serving explicit fossil shadow aliases such as /path/to/file.cgi from /path/to/file.cgi.html as a fail-loud fallback when no service worker controls the route.

Canonical Rails-ish Flow

// service-worker.js
self.onfetch = (event) => {
  const url = new URL(event.request.url)
  if (event.request.method === "POST" && url.pathname.endsWith("/posts/create.cgi")) {
    event.respondWith((async () => {
      const form = await event.request.clone().formData()
      const params = new URLSearchParams(form)
      return Response.redirect(new URL("./create.cgi.html?" + params, event.request.url), 303)
    })())
  }
}

// posts/create.cgi.html, usable during initial rendering
const posted = new URLSearchParams(location.search)
document.write(posted.get("post[title]") || "Untitled post")

Fossil Dynamic Routes

  • On-disk dynamic extensions are never executable and are not a plugin API: .cgi, .pl, .php, .asp, .aspx, and .jsp are respected and feared.
  • If path/to/file.cgi.html exists, path/to/file.cgi may resolve to that static HTML shadow page.
  • If no shadow page exists, Web.app must return a clear error explaining that server code is unsupported and naming the expected shadow file.
  • No compatibility feature should silently swallow an unsupported method, missing shadow page, parse failure, or unsafe request.
  • Server Side Includes are not enabled as a blanket templating language; add only the smallest static include behavior needed by a checked-in example bundle.

Native Origin State

  • With stable *.localhost:55555 origins, Web.app should use WebKit's persistent website data store so native origin state survives reopening.
  • Service worker registrations, CacheStorage, IndexedDB, cookies, OPFS, and related origin state should be treated as supported web platform features.
  • Do not duck-punch supported native APIs merely to police storage. Warnings are appropriate only for actually unsupported features or transitional builds.
  • File-backed localStorage remains a Web.app enhancement for document files, not a reason to disable normal web storage.
  • Native origin state is isolated by webspace origin; filesystem writes still require the strict localStorage file-key path grammar from the stable-origin spec.

Example-Driven Compatibility

  • Do not add compatibility behavior without a concrete examples/web-compat/*.web bundle that needs it.
  • Use cartoonishly idiomatic examples first: guestbook, early AJAX suggest, Rails-ish weblog, and later PWA/offline fixtures.
  • Each fixture should document the era convention, the exact native web mechanism used, and the safety boundary.
  • After each red fixture passes, refactor only if the shared behavior becomes simpler, more general, and no less safe.
  • Normal browsers should degrade by feature detection: if service workers, Navigation API, or persistent storage are unavailable, the page should still present a clear static fallback.

Implementation Notes

  • AppifyHost should load Web.app through stable localhost origins that satisfy secure-context requirements for service workers and other modern APIs.
  • Web.app should serve worker scripts as ordinary JavaScript files, respect normal service worker scope rules, and avoid injecting page-only scripts into worker responses.
  • A bundle-root service-worker.js controls its own .web path by default; broader webspace control requires a worker at the broader path, not a hidden host override.
  • Existing POST shadow injection may remain only as a migration fallback until examples are converted to service-worker-owned request handling.
  • The 2005-rails-weblog.web fixture should register a service worker and use POST-to-303-to-query for its create/comment flows.
  • All failures that matter to humans should be visible: console warning for progressive enhancement gaps, HTTP error page for unsupported server behavior, and in-page message for direct shadow-page opens.

Acceptance Tests

  • window.isSecureContext is true for a Web.app .web bundle.
  • A bundle can register service-worker.js; after activation, a form POST is observed by the worker as a native Request.
  • The Rails weblog example submits to posts/create.cgi, the service worker returns a 303 redirect, and create.cgi.html renders from location.search without Appify globals.
  • Feature-detected NavigateEvent.formData code runs where WebKit supports it and gracefully falls back where it does not.
  • AppifyHost and __WEB_APP_REQUEST__ are absent from updated web-compat examples.
  • A missing fossil shadow page fails with a loud HTTP error; an existing .cgi.html shadow page is served without executing server code.
  • Service worker registration, CacheStorage, and IndexedDB persist across Web.app restarts for the same stable webspace origin.

Golf Rationale

Option Technical Liability Verdict
Real service workers plus Navigation API Lowest bespoke surface, highest browser-native smell, and naturally useful outside Web.app. Default.
Host-injected window.AppifyHost.request Small short-term implementation, but creates an Appify-specific programming model and teaches the wrong abstraction. Migrate away.
Synthetic window-level fetch events Looks like service workers but violates where the platform actually places request interception. Reject.
Fake REST/server emulation Quickly becomes uncanny, insecure, and unbounded without real examples. Only add concrete red-case behavior.
