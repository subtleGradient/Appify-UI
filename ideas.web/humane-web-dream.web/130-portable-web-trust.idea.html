Portable Web Trust

This is a research note, not an approved proposal. The question is how Web.app can support decentralized storage, identity, provenance, and offline-first sharing without adopting finance-coded wallet language or pretending that "Web3" is a native browser-platform era.

start / values / decisions / portability / uncertainties / next probes

Current Stance

  • Avoid the term Web3 in product language; use portable web, verifiable webspace, or decentralized web only when the technical mechanism matters.
  • Respect pro-crypto and anti-crypto users by not forcing the argument into the interface.
  • Do not add window.ethereum, wallet prompts, chain RPC, smart contracts, tokens, or transaction signing to Web.app core.
  • Do not block authored bundles from using ordinary JavaScript libraries, remote RPC, WalletConnect-style flows, or chain-specific integrations when the author chooses them.
  • Put durable value in web-native affordances: stable origins, service workers, CacheStorage, IndexedDB, OPFS, Web Crypto, relative links, static files, and visible provenance.

Standards Terrain

Layer Status Web.app Implication
Browser platform WHATWG standards cover HTML, DOM, URL, Fetch, Streams, Storage-related platform pieces, and other living web foundations. There is no WHATWG "Web 3.0" platform version. Keep Web.app boringly compatible with normal browser APIs. Do not invent Appify-only page globals for decentralized features.
Decentralized identity DID Core is a W3C Recommendation; Verifiable Credentials Data Model v2.0 is also a W3C Recommendation. Treat identity and credentials as optional provenance and verification metadata, not as wallet login by default.
Browser-mediated credentials Digital Credentials API is draft-stage web standards work. Feature-detect later. Do not make current Web.app behavior depend on it.
Content addressing IPFS CIDs, CAR files, and IPNS are real decentralized-web conventions, not WHATWG browser APIs. Useful as import/export and source-materialization formats if Web.app keeps normal local HTTP origins and verifies bytes before serving them.
Wallet providers EIP-1193 defines an Ethereum provider interface; browser injection such as window.ethereum is ecosystem convention, not native web law. Keep this out of core. If needed, make it an explicit Web.app extension or separate chain-oriented host.

Fit With Existing Decisions

  • Native Web Compatibility already says real service workers, normal Request/Response, and persistent browser state are the foundation.
  • Stable Webspace Origins already gives each webspace a secure, persistent, same-origin place to run browser-native decentralized libraries.
  • Portability already says the same bytes should work as a local document, static website, emailed zip, removable-media archive, or future mobile bundle.
  • Decisions currently says no decentralized identity protocol, registry, or sync layer yet; this note keeps those as open research, not hidden commitments.

Candidate Surfaces

Surface Smallest Useful Shape Failure Mode To Avoid
Content-addressed source Add a future .web manifest source kind that materializes a verified CAR/CID into a read-only local cache, then serves it through the normal stable webspace origin. Letting a remote resolver become a hidden runtime dependency for core document behavior.
Signed provenance Allow an optional manifest or sidecar that says which bytes were verified, who signed them, and which key or DID document was used. Implying real-world trust from a cryptographic signature alone.
Mutable names Resolve IPNS, DNSLink, DID documents, or other mutable pointers only into pinned immutable content before opening. Serving moving remote content as though it were the durable local artifact.
Capability sharing Explore UCAN-like or capability-token grants for document sharing, scoped to a webspace and readable as plain metadata. Creating an invisible account system or authority service that users cannot inspect.
Chain-specific affordances Leave room for an explicit extension that can provide EIP-1193 or other wallet/provider APIs to opted-in bundles. Making Web.app feel like a crypto wallet or requiring every user to understand accounts, gas, tokens, or chains.

Suggested Probe

Build a tiny read-only fixture: a .web manifest that points at a content-addressed static bundle, materializes the bytes into a read-only cache, verifies the expected root address or digest, and opens the result at a normal *.localhost:55555 Web.app URL. No wallet, no account, no chain, no token, no live remote requirement after materialization.
