Service Worker Terrain

Service workers are browser-managed, origin-scoped event runtimes. They can make a static web bundle feel like a small local web appliance, but they also create durable browser state that can outlive old deployments, old build systems, and old cleverness.

start / portability / uncertainties / next probes / portable web trust

Current Stance

  • Read service workers as local HTTP-shaped browser machinery, not as a hidden server.
  • Separate the durable registration from the active worker version and from the temporary JS runtime.
  • Assume old users may carry old registrations, old script URLs, old cache names, and old IndexedDB state.
  • For Web.app, service workers are powerful enough to deserve explicit terrain notes before they become invisible infrastructure.

Object Map

Thing Where It Lives What It Means What Can Go Wrong
scriptURL Server URL such as /sw.js. The canonical URL the browser fetches to install or update a worker version. If that old URL later 404s, the browser may be unable to install a cleanup replacement.
Registration Browser profile storage, keyed by storage partition, origin, and scope. The durable mapping from scope URL to installing, waiting, and active worker versions. It persists after tabs close and can survive many app releases.
Worker version Registration slots: installing, waiting, active. One installed incarnation of the worker script. Old active and new installing/waiting versions can overlap and touch the same durable storage.
Runtime A temporary ServiceWorkerGlobalScope JS universe. The actual running heap and event loop created when events need delivery. Globals are not durable. They vanish when the runtime is stopped and rebuild on the next wake.
Cache Storage / IndexedDB Browser origin storage. Durable data shared by worker versions and controlled pages. Version races and old migrations can leave durable state newer or older than the active code expects.

Universe Created By DOM? Main Crossing Lifetime Shape

What Register Means

navigator.serviceWorker.register("/sw.js") asks the browser to create or update a registration for a scope. First time, the browser fetches the script, creates a ServiceWorkerGlobalScope, evaluates top-level code, fires install, then later activate. Later calls return the existing registration unless the browser discovers that the worker script changed. 

page calls register("/sw.js")
  -> browser resolves script URL and scope
  -> browser fetches /sw.js
  -> browser evaluates top-level sw.js in ServiceWorkerGlobalScope
  -> install event
  -> waiting slot if old clients still exist
  -> activate event
  -> active slot controls matching future clients

MDN documents register() as creating or updating a service worker registration, and the Service Workers specification defines the registration, job, update, install, and activation algorithms.

Script URL, Scope, And Header

Piece Meaning Edge
scriptURL Where the browser gets the worker code. Updates keep checking this exact URL, including old hashed or deep paths.
Default scope The script URL's directory. /a/b/sw.js defaults to /a/b/. Folder-like only because the default ends in a slash.
Explicit scope A URL parsed relative to the registering page's base URL. { scope: "foo" } from /a/page.html becomes roughly /a/foo, not relative to the worker script.
Service-Worker-Allowed Response header on the worker script that broadens the maximum allowed scope. A deep script cannot claim / unless its response permits it. 
GET /old/deep/sw-2021.js
Content-Type: text/javascript
Service-Worker-Allowed: /

// This deep script may now be registered for scope "/".

The header is documented at MDN Service-Worker-Allowed. It belongs to the worker script response, not the HTML page.

Scope Matching

Scope matching is serialized URL prefix matching. If multiple registrations match one client URL, the longest matching scope wins. This is not a nested layout cascade. Parent and child service workers do not both run for the same page.

Stored Scope Page URL Result
https://site.test/ https://site.test/path/deep/page.html Matches.
https://site.test/path/deep/ https://site.test/path/deep/page.html Matches and beats root because it is longer.
https://site.test/app https://site.test/apple Can match because the serialized URL starts with /app.
https://site.test/page.html?v=1 https://site.test/page.html?v=123 Can match because ?v=123 starts with ?v=1.
https://site.test/page.html?v=1 https://site.test/page.html?x=0&v=1 Does not match because query order changes the serialized prefix.

Control Does Not Require Fetch Handling

A controlled page is a Window client associated with the active registration selected by its URL. The worker can control a page even if it never registered a fetch handler. In that case normal requests go through the normal network path, but navigator.serviceWorker.controller can still exist and message the active worker. 

page URL is inside winning scope
  + active worker version exists
  + page was born under that registration
  = navigator.serviceWorker.controller is non-null

Fetch Crossing

A controlled page does ordinary web things: navigation, images, scripts, styles, forms, fetch(), and XHR. The browser's fetch machinery may deliver those requests as FetchEvent objects to the active service worker runtime. 

controlled page makes Request
  -> selected service worker receives FetchEvent
  -> handler may call event.respondWith(responsePromise)
  -> otherwise request proceeds on the normal path
/*
  globalThis here is ServiceWorkerGlobalScope, not Window.
  This assignment runs during top-level script evaluation.
*/
globalThis.onfetch = event => {
  /*
    event.request may represent navigation, CSS, JS, image, form, fetch(), or XHR.
  */
  if (event.request.mode !== "navigate") return;

  /*
    respondWith() is the point where this worker supplies the HTTP-shaped answer.
  */
  event.respondWith(fetch(event.request));
};

Move 0: Top-Level Script Evaluation

Before an event can be delivered, the browser must have a running service worker runtime. If one is not already alive, the browser creates a fresh ServiceWorkerGlobalScope, evaluates top-level sw.js, then dispatches the waking event. 

request needs worker
  -> no runtime alive
  -> create ServiceWorkerGlobalScope
  -> evaluate sw.js top-level
  -> var abc = doExpensiveThing()
  -> globalThis.onfetch = handler
  -> dispatch FetchEvent

Top-level work runs once per runtime start, not once forever and not necessarily once per request. If the browser stops the runtime between events, top-level work runs again on the next wake. A huge global object lives only in that temporary heap; durable data belongs in IndexedDB, Cache Storage, or another storage surface.

Lifecycle States

[new bytes found]
  -> installing
      -> install succeeds
          -> waiting
              -> old controlled clients go away, or skipWaiting()
                  -> activating
                      -> activated
  -> install fails
      -> redundant

old active worker
  -> remains active while new worker waits
  -> becomes redundant after replacement
State Meaning Edge
installing New script version is being admitted. If top-level evaluation or install lifetime work fails, the candidate is discarded.
waiting New worker installed, but old active clients still exist. This prevents a page from being half-controlled by old and new worker code by default.
active The version that controls matching future clients. active.state does not prove the runtime is currently awake.
redundant A version that has failed or been replaced. Durable storage it wrote may remain.

Versioning And Updates

The browser's service-worker versioning is based on fetching the registration's scriptURL and checking whether the script resources differ. Your own CACHE_VERSION string is only application data unless the served worker bytes actually change.

Mechanic What Happens Failure Mode
Same bytes No new worker version installs. Changing only cached assets does not update the worker brain.
Different bytes New version enters installing. New code may wait behind old controlled pages.
importScripts() Imported scripts participate in update behavior, with cache use affected by updateViaCache. A CDN-stale import can keep old logic alive or split populations.
updateViaCache Controls whether HTTP cache is consulted for the main script and imports during update checks. all can make update checks more cache-shaped than expected.
skipWaiting() Moves a newly installed worker past the waiting ledge. Old page JS can suddenly speak to new worker logic.
clients.claim() Active worker claims currently open in-scope clients. Pages that were born uncontrolled can change controller mid-life.

See MDN updateViaCache and the specification's update algorithm for the exact update-check surface.

Overlapping Versions

active v1 handles current pages
installing v2 runs top-level code and install event
v1 and v2 have separate heaps
v1 and v2 can both touch Cache Storage and IndexedDB
v2 waits until old clients leave, unless skipWaiting() is called

This is the central race shape: separate JS universes, shared durable storage. The browser serializes some lifecycle jobs, but it does not make application-level cache names, database migrations, or old/new response formats coherent.

Hyperslice Of Cute Failure Modes

Razor Half A Half B Over-Clever Failure
Scope shape Folder-like /app/. Prefix-weird /app, ?v=1. /apple or ?v=123 unexpectedly matches.
Scope depth Root scope. Nested scope. Deepest matching registration wins, so root cleanup can miss deep registrations.
Script URL Stable /sw.js. Hashed, generated, or per-user URL. Old users keep checking a dead old URL forever.
Response source Network-first. Cache-first app shell. New cleanup JS never reaches the page because old HTML is served from cache.
Activation Wait for old clients. skipWaiting(). Mixed old page runtime with new worker responses.
Claiming Next navigation only. clients.claim(). Current open pages suddenly become controlled.
Storage ownership One version writes. Old and new versions both write. Migrations, cache deletes, and response schemas race.
Fetch policy Narrow routes. Catch-all. APIs, OAuth callbacks, logout, HTML, assets, and forms receive the same fallback.
Recovery path Old script URL still served. Old script URL gone. The browser cannot fetch the cleanup replacement for that registration.

Inspection

From any fresh page that actually reaches new code, inspect the registration table for the current origin and storage partition: 

/*
  getRegistrations() returns this origin's service worker registrations
  visible to the current storage partition.
*/
navigator.serviceWorker.getRegistrations().then(registrations => {
  console.table(registrations.map(registration => ({
    scope: registration.scope,
    installing: registration.installing && registration.installing.scriptURL,
    waiting: registration.waiting && registration.waiting.scriptURL,
    active: registration.active && registration.active.scriptURL
  })));
});

/*
  controller is only the winning active worker for this page URL.
*/
console.log(navigator.serviceWorker.controller && {
  scriptURL: navigator.serviceWorker.controller.scriptURL,
  state: navigator.serviceWorker.controller.state
});

Chrome exposes registration status, source, start/stop, update, unregister, and bypass controls under DevTools Application > Service Workers. See Chrome DevTools PWA debugging.

Server Log Recovery Signals

If new page code cannot load because an old worker serves an old app shell, server logs may be the only primitive mark. Watch for missing worker script requests: 

GET /sw-2021.js                 404
GET /assets/sw.abcd1234.js      404
GET /old/pwa/service-worker.js  404
GET /app/sw.js?v=17             404

Do not rely on visible redirects for worker-script recovery. The update fetch for the top-level service worker script treats redirects as failure in the specification. Serve a real JavaScript response at the exact old URL instead.

Cleanup Worker Shape

For each old scriptURL, internally rewrite that exact URL to a cleanup worker response. Keep the browser-visible URL unchanged. 

HTTP/1.1 200 OK
Content-Type: text/javascript; charset=utf-8
Cache-Control: no-cache, no-store, must-revalidate
Service-Worker-Allowed: /

/*
  This runs in ServiceWorkerGlobalScope, not Window.
  It replaces an old worker at this exact scriptURL, then unregisters.
*/
globalThis.oninstall = event => {
  /*
    skipWaiting() moves this cleanup version past the waiting ledge.
  */
  globalThis.skipWaiting();
};

globalThis.onactivate = event => {
  /*
    unregister() removes this scope registration from browser storage.
  */
  event.waitUntil(globalThis.registration.unregister());
};
Server Response Likely Result
200 valid JavaScript at old URL. Browser can install cleanup replacement.
301 or 302. Update can fail because redirects are not accepted for the top-level worker script.
HTML fallback. MIME check fails; old active worker may remain.
404. Update fails; old active worker may remain.
Missing broad enough Service-Worker-Allowed. Broad old scopes may fail update with the cleanup script.
