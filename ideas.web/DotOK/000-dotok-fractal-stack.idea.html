DotOK Fractal Stack

DotOK should be a specialized IDE built from smaller reusable document nouns, not a monolith. WebForm handles structured visible input. The generic noun layer handles file shapes, kind servers, and URI glue. DotOK owns the homeostatic ontology on top.

Dependencies: single-file webform / composable document nouns

DotOK-Specific Capture

Double-click a .ok bundle and open it in a new DotOK.app document window.

Inside a DotOK bundle are many things. Each thing has zero or more sub-things, which are also things. Each thing has many aspects.

Each aspect has one way in which it can be good, healthy, and stable: its homeostatic balance point, the needle hole to thread. It may have many bad failure modes: too high, too low, too far left or right, the wrong material, incomplete passage, stale evidence, and similar misses.

Assessments are point-in-time reports about how reality is now. Future snapshots are not-yet-real slices used to orient goals before concrete work orders exist.

Dependency Stack

Layer Owns DotOK Uses It For
.webform Single-file visible HTML form state and WebFormer rewrite/save behavior. Balance-point forms, assessment prompts, future-snapshot prompts, review packets.
Composable nouns Single-file versus bundle rules, kind servers, URI glue, fail-closed composition. Opening each DotOK noun in the correct local view without a giant shared tool.
.canvas Graph-shaped standard format. Homeostatic maps, focus slices, thing/aspect graphs, work-order dependency maps.
.appdev Workshop around a shippable app. Designing DotOK.app while keeping private process outside the runnable app bundle.
DotOK ontology Things, aspects, balance points, failure modes, assessments, futures, goals, work orders. The specialized IDE experience that sits above the generic layers.

DotOK Nouns

Noun Meaning Likely Representation
thing Any object of care. A thing may contain sub-things, which are also things. things/*.thing/thing.json5 plus child refs.
aspect A dimension along which a thing can be healthy, drifting, broken, ambiguous, or unknown. aspects/*.aspect/aspect.json5, often with a balance-point .webform.
balance point The one homeostatic way the aspect is good: the needle hole to thread. A visible .webform, assertion, evaluator, or canvas node owned by the aspect.
failure mode One recognizable way to miss the balance point. failure-modes/*.mode.json5 with examples and detection notes.
assessment A point-in-time reality snapshot with frozen observed values and evidence. assessments/YYYYMMDD-HHMM.assessment/.
future snapshot A desired not-yet-real target slice. It is not a plan and does not own tasks. futures/*.future/ with focus canvas and target-state webforms.
goal A high-level desired outcome derived from a future snapshot or assessment gap. goals/*.goal.json5.
work order An executable unit that can unblock goals or other work orders. work-orders/*.work.json5 or compatibility with existing .tasks/*.md.

Important Naming Tension

The current DotOK protocol already uses <FOCUS_ROOT>/.ok/ as the hidden evergreen ideal-state directory. A Finder document named My Project.ok is different, but the conceptual overlap is sharp enough that the first implementation needs an explicit bridge.

  • Option A: .ok becomes the visible Finder package for new DotOK documents, while legacy project roots keep hidden .ok/.
  • Option B: .ok stays the hidden protocol directory and the Finder package uses .dotok or .okbundle.
  • Option C: DotOK.app opens both visible *.ok/ packages and regular project folders with hidden .ok/ and .tasks/.

Likely Build Sequence

  1. Build the standalone .webform fixture and WebFormer rewrite loop first.
  2. Prove the generic kind-server and URI pattern on .webform and .canvas.
  3. Create a toy needle-threading.ok bundle with one thing, one aspect, one balance point, several failure modes, one assessment, one future, one goal, and two work orders.
  4. Prototype DotOK.app opening the .ok bundle and dispatching each noun to its kind-specific local server.
  5. Only then start designing the specialized DotOK IDE views.

DotOK Open Decisions

  • How much snapshot content is copied versus content-addressed. Assessments need to stay stable when live definitions evolve.
  • Whether failure modes are global reusable patterns, aspect-local children, or both.
  • Whether future snapshots become immutable after goal generation or remain editable with versioned revisions.
  • How work orders map to the existing DotOK .tasks/*.md YAML protocol without creating two task systems.
  • How dotok:/// handles moved bundles, multiple open documents, URL encoding, permissions, and cross-document links.
  • How the IDE stays cognitively quiet when the underlying graph becomes enormous.
