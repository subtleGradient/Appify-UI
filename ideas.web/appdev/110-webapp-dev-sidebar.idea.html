Webapp Dev Sidebar

Expo proves that .webapp is not always a pure browser loop. Some package dev servers are also terminal UIs, so Webapp may eventually need a native split between lifecycle output and the rendered local URL.

Sequence: seed / web bundle composition

Idea

  • Keep today's default: Webapp.app opens the first loopback URL printed by bun dev.
  • Add an optional macOS sidebar later, not a mandatory IDE shell.
  • The first sidebar should be a read-only lifecycle log backed by .local/dev.log.
  • Promote to an interactive terminal only for packages that prove it is necessary, such as Expo's Terminal UI commands.
  • Use standard macOS sidebar/window chrome so the live web view remains the primary document surface.

Possible Shape

  • Web: the rendered stable-origin page.
  • Dev: lifecycle log, install/dev status, ready URL, and restart button.
  • Terminal: optional TTY view only when the package asks for it or Webapp detects a known TUI-heavy dev server.
  • Files: defer. A file browser would pull Webapp toward IDE gravity too early.

Thing Golf

  • Read-only log sidebar: Safety 0, Burden 1, Chaos 0, Betrayal -1, Control 0. The main value is honest lifecycle visibility.
  • Interactive TTY sidebar: Safety 1, Burden 2, Chaos 1, Betrayal 0, Control 2. Useful only if Expo-style terminal commands become common enough.
  • File explorer sidebar: Safety 1, Burden 2, Chaos 2, Betrayal 0, Control 3. Defer because it changes the app from document host to IDE shell.

Next Probe

  • Open the Expo example and inspect whether .local/dev.log is enough.
  • If not, prototype the smallest read-only sidebar against existing Webapp logs.
  • Do not add terminal input until a package needs actual TUI commands after launch.
