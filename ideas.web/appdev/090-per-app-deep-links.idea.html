Per-App Deep Links

AppifyHost links should be app-specific. Web.app owns dotweb:; future host apps own their own schemes instead of sharing one global appify: router.

Sequence: seed / macOS conventions / bundle taxonomy

Decision

  • Web.app registers dotweb: through CFBundleURLTypes.
  • Deep links are an explicit ingress path, not a relaxation of normal WebView navigation.
  • The initial public shape is dotweb://open?document=/absolute/path/Site.web&route=/page.html and dotweb://choose?route=/page.html.
  • Opening a raw local document path from a deep link requires user confirmation, because websites can launch custom schemes.

Why Per-App Schemes

  • Custom schemes are global names. If two apps claim the same scheme, the operating system does not give a clean ownership model.
  • Each AppifyHost app has its own document extension, app-local command vocabulary, and trust boundary.
  • A shared appify: scheme would turn one router into a privileged broker for every personal app, which is the wrong default for hackable local software.

Linux Future

  • No Linux implementation yet.
  • When AppifyHost gets Linux support, each host app should register its own x-scheme-handler/* MIME association through a .desktop file.
  • For Web.app, the desired handler is x-scheme-handler/dotweb.
  • Arch + Wayland should be treated as an XDG/portal integration problem, not a Wayland protocol problem: xdg-mime, mimeapps.list, and xdg-desktop-portal are the relevant future surfaces.

Sources To Recheck Before Linux Work

  • Desktop Entry Specification for MimeType=x-scheme-handler/dotweb;.
  • MIME Applications Associations Specification for defaults in mimeapps.list.
  • OpenURI portal for sandboxed browser/app handoff.
