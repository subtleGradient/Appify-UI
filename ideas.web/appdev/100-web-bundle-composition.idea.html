Web Bundle Composition

.web bundles compose through ordinary relative web URLs. The host gives them a stable local webspace, but readable bytes still come only from opted-in .web packages.

Sequence: seed / per-app deep links

Decision

  • Use native web imports, CSS links, anchors, and fetch() between packages. Do not invent a custom URL scheme for module loading.
  • If the opened package is inside a normal git repo, use the nearest non-home, non-root git worktree root as URL geometry.
  • If there is no normal git root, use the opened package's parent directory so sibling .web packages can compose.
  • A .web may be either a directory bundle or a JSON5 file manifest. Empty files and empty folders are valid creation affordances, but Web upgrades them on first open.
  • Only contents inside .web packages are readable through the local webspace. Random repo files are not web-addressable.

Portable Shape

project/
  apps/
    dashboard.web/
      index.html
  packages/
    ui-kit.web/
      exports/button.js
      exports/tokens.css
<link rel="stylesheet" href="../../packages/ui-kit.web/exports/tokens.css">
<script type="importmap">
{
  "imports": {
    "ui-kit/": "../../packages/ui-kit.web/exports/"
  }
}
</script>

Storage Boundary

  • The opened package owns the window's duck-punched localStorage.
  • Imported modules from a peer package run as library code inside the caller's storage context.
  • Fetched peer HTML is served raw, without injected storage or live reload scripts.
  • Opening a peer .web package creates a separate window with separate storage.

File Manifests

  • touch Site.web becomes a JSON5 file manifest with a build-pinned $schema.
  • mkdir Site.web becomes a starter static bundle by copying Web's bundled Untitled.web template.
  • @/ in a manifest means the nearest real git root; without git, generated manifests use ./.
  • Commit-pinned GitHub sources can mount a repo subpath at the manifest's normal .web URL route without submodules or a CDN dependency.

Golf

  • Low badness: this follows normal static web deployment and avoids CORS, registries, custom fetch schemes, and build steps.
  • Manifest files are only for places where convention cannot carry the source location: blank file upgrades, local aliases, and commit-pinned remote repo subpaths.
  • The git-root rule is URL geometry only. The opt-in safety boundary is still the .web package.
  • Ignoring ~/.git and /.git avoids accidental whole-home or whole-disk webspaces without adding a policy framework.
