AI Chat and Memory Apps

Appify UI is already good at turning local artifacts into small native document apps. The next interesting lane is AI work memory: file-based .chat bundles, sharded SQLite conversation archives, and project/topic packages that keep chats, notes, prompts, assets, and generated context together.

Atlas links: index / method / host patterns / prototype backlog

Observed Local Shapes

Readable .chat Folders

The current file-based chats are plain directory packages with numbered Markdown turns such as 001-user.md, 002-model.md, optional *-thought.md, model parts, and usage sidecars. They are excellent project artifacts because Git, editors, grep, and Codex can all understand them.

Queryable .chat/db Archives

The SQLite archive shape has a catalog.db, yearly conversation shards, actors, events, parts, assets, import batches, and source metadata. It is the right home for thousands of imported conversations, but it wants a focused browser instead of a terminal query loop.

Project Memory Packages

The useful human object is often bigger than one conversation: a project or topic bundle containing selected chats, notes, prompts, generated outputs, assets, maps, and indexes. That object should remain a normal Finder package.

Hyperslice Map

Artifact vs archive Readable project package Queryable long-term store Do not force one format to win. Build bridges: archive search can export curated .chat or project packages.
Viewer vs workbench Open and inspect safely Reshape, annotate, export, rerun First app should be viewer-first with explicit save/export operations. Later apps can own heavier mutation.
Conversation vs memory One transcript Project/topic context object Individual chat viewing is necessary, but the higher-leverage app is a bundle that organizes many chats plus notes.
Human reading vs agent fuel Pleasant transcript UI Context-pack generation Render cleanly for humans, then provide export modes for LLM handoff, project resume, decision extraction, and prompt packs.
Timeline vs topology Chronological conversation list Topic, decision, task, and reference graph Start with timeline/search. Add graph/canvas views once links and tags have a durable package shape.
Private by default vs shareable slice Local full-fidelity memory Redacted export package Any sharing flow needs an explicit redaction/export boundary, not silent mutation of the source archive.

App Ideas

App Opens First behavior Why it fits Appify UI
ChatDock.app .chat folders Render numbered Markdown turns as a transcript; collapse thoughts; show model parts and usage; edit/save Markdown turns. Closest match to existing document package apps. It makes a real local artifact double-clickable without inventing a new archive.
ChatArchive.app .chat archive roots or .chat/db Browse/search catalog conversations; filter by source, shard, date, actor, and asset count; open one conversation detail view. Turns the sharded SQLite archive into a native-feeling archive browser while keeping SQLite as the canonical store.
ProjectMemory.app .memory or .chatpack packages Organize selected chats, notes, prompts, assets, decisions, and open questions by project or topic. Promotes the object that actually matters in daily work: a bounded local memory bundle for one project.
ContextForge.app Repo folders, .chatpack, or .memory Select chats, notes, repo files, canvases, and transcripts; emit LLM-ready context packs and handoff briefs. Bridges human curation and agent fuel without making the chat viewer responsible for every workflow.
ConversationMap.app .chat, .chat/db, or .memory Show chats, topics, decisions, tasks, prompts, and artifacts as a graph/canvas with typed links. Natural sibling to JSONCanvas/TLCanvas once there is enough metadata to justify a topology view.

Minimal Package Sketch

Keep the first package boring and inspectable. It should feel like a Finder folder first and a database second. 

my-topic.chatpack/
  index.json
  chats/
  notes/
  prompts/
  assets/
  exports/
  • index.json names the package, project/topic, source roots, pinned chats, and generated exports.
  • chats/ contains file-based .chat folders, copied or linked from archive exports.
  • notes/ and prompts/ stay plain Markdown so Codex, grep, and regular editors remain first-class.
  • exports/ holds generated context packs, briefs, redacted bundles, and one-off handoffs.

Recommended Build Sequence

  1. Build ChatDock.app as a viewer/editor for one file-based .chat directory.
  2. Add read-only open support for a .chatpack containing many .chat folders and Markdown notes.
  3. Build ChatArchive.app against the existing conversation-store service and let it export selected archive conversations into .chatpack.
  4. Promote ProjectMemory.app once the package conventions are stable enough to carry decisions, prompts, and generated context.
  5. Split off ContextForge.app and ConversationMap.app only after the underlying metadata exists outside the UI.
