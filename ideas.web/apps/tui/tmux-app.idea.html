paused TUI idea / do not generalize yet

tmux.app

A native Mac shell for active tmux sessions is attractive, but its lifecycle model is not the same as the current AppifyHost document model.

Atlas links: TUI top picks / devops / host patterns

Desired Behavior

  • Launching tmux.app restores all active tmux sessions from the default tmux socket.
  • Each active tmux session appears as its own app window.
  • If a new tmux session is created from Terminal while tmux.app is running, the app notices and opens a new window for it.
  • Closing a tmux.app window kills that tmux session.
  • Quitting or killing tmux.app does not kill tmux sessions; it only detaches the app's clients.

Shared Host Boundary

Rule: if only one app needs a behavior, it does not go into shared AppifyHost code yet.

Need Current evidence Decision for now
Discover external runtime objects with no document path Only tmux sessions clearly need this. Keep tmux-specific.
Mirror every external object as a native window Existing apps are one opened file, folder, or package per window. Do not add a generic host window protocol yet.
Watch for new objects while the app runs Possible future siblings are screen.app, zellij.app, or dtach.app, but those are tmux-shaped siblings, not independent AppifyHost proof. Prototype inside tmux.app first.
Send window-close lifecycle events back to a supervisor The close semantics are app-specific: close kills session, app quit detaches. Avoid shared host API until another app needs the same semantics.

Non-Examples

  • LazyGit.app, tw.app, litecli.app, and LogScope.app are still one opened object to one server to one window.
  • TLCanvas.app, JSONCanvas.app, WebFormer.app, and WikiDock.app can put tabs, panes, or subviews inside their own web app rather than asking the native host to spawn runtime-discovered windows.
  • Potential profile apps like KubeContext.app, RemoteFiles.app, APIWorkbench.app, and Runbook.app still start from an explicit profile, package, or file document.
  • Future LogScope saved sessions or log bundles might grow richer state, but that should be proven as a LogScope package feature before becoming host behavior.

Prototype Shape

  • Build tmux.app as a bespoke app bundle or tmux-specific Swift/AppKit host.
  • Use plain tmux commands and ttyd tmux attach-session before considering tmux control mode or a custom terminal renderer.
  • Track only the default tmux socket in v1, matching plain tmux list-sessions.
  • Promote a shared AppifyHost abstraction only after a second non-tmux app proves the same lifecycle shape.
