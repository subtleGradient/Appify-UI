Prototype Probes

The next implementation should prove Webapp.app before any SDK app becomes a product. The falsifier is simple: if normal packages need too much Appify ceremony, the abstraction is not generic enough.

Read order: index / problem / webapp vs website / jobs / Webapp first / layers / package contract / dependencies / SDK profiles / golf

Probe Order

Order Probe What it proves Stop condition
1 Webapp.app opens an empty .webapp and scaffolds a normal local web project. The base AppifyHost content-package mode and runner are enough for generic web apps. A blank webapp needs SDK-specific assumptions before it can render.
2 d3-bar-chart.webapp opens with no package-local node_modules. Normal package entry and host SDK aliasing work. Common D3 examples need Appify-specific rewrites before they render.
3 small-network.webapp opens and saves graph.cyjs. Data-first SDK profile works without a dedicated Cytoscaper.app. Saved output loses style, layout, or graph data compared with cy.json().
4 two-node-flow.webapp opens built-in nodes and then one custom node component. Custom renderer ceiling works without abandoning normal package layout. Custom nodes require a bespoke build path per package.
5 json-rete.webapp opens JSON-valid nodes first, then a registered class-based node. The profile can distinguish safe JSON from serializer-required objects. The host cannot produce clear errors for missing serializers.
6 snapshot-canvas.webapp opens a tldraw document/session split. The generic contract can absorb TLCanvas-like packages without flattening them. Existing TLCanvas semantics become less clear or less inspectable.
7 Export one .webapp example as a static .website and reopen it without npm/cache/build support. The static bundle boundary is real. The exported package still needs package.json, dependency resolution, or a build step.
8 Reassess whether .d3, .cytoscape, .xyflow, .rete, or .tlcanvas still deserve dedicated apps. Dedicated apps are justified by evidence, not by SDK names. Every dedicated app would only wrap the same .webapp behavior.

Fixture Packages

  • hello-webapp.webapp: generic empty/new package scaffold.
  • d3-bar-chart.webapp: one index.html, one src/main.js, one CSV file.
  • small-network.webapp: one graph.cyjs, one stylesheet, one layout config.
  • two-node-flow.webapp: built-in nodes only, then a custom node variant.
  • json-rete.webapp: plain JSON nodes and connections.
  • snapshot-canvas.webapp: tldraw document/session split plus assets.
  • d3-bar-chart.website: static export with no npm or build-step expectation.

Pass Bar

  • Open within the existing AppifyHost startup timeout or a profile-specific justified timeout.
  • Render useful content before any dependency install prompt.
  • Save a no-op document without diff churn.
  • Delete the cache and reopen from package files.
  • Copy the package to another folder and reopen with the same result.
  • Static .web and .website exports must reopen without package-manager access.
