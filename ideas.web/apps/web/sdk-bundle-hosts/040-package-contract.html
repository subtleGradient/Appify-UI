Package Contract

The package should be a normal .webapp first, and Appify-specific only when needed. SDK-specific extensions are later aliases for packages whose examples prove they deserve a separate Finder identity.

Read order: index / problem / webapp vs website / jobs / Webapp first / layers / dependencies / SDK profiles / golf / probes

Three Levels

Level Shape Use when App behavior
0. Normal package package.json, index.html, src/, data/, public/. A small web project already exists. Infer entry and resolve host SDK aliases.
1. Appify annotated Normal package plus appify.json. The package needs a canonical save file, custom entry, permissions, or migration info. Use explicit contract and show warnings for missing capabilities.
2. Sealed package Annotated package plus vendor/ or deliberate node_modules/. Offline, archival, classroom, lab, or reproducible handoff matters more than size. Prefer package-local deps and never mutate them silently.

.web / .website Contract

  • .webapp may contain package.json, lockfiles, source transforms, and npm dependency declarations.
  • .web and .website must not require package.json, node_modules, npm install, TypeScript compilation, JSX transform, or any bundler at open time.
  • .web and .website are static web-native packages: index.html, CSS, browser JavaScript, assets, data, and relative links.
  • A .webapp may export a static .website, but the export must stand alone as static web tech.

Small appify.json

{
  "appify": 1,
  "kind": "webapp",
  "profile": "d3",
  "entry": "./src/main.ts",
  "document": {
    "canonical": ["./src", "./data"],
    "session": "./state/session.json",
    "preview": "./QuickLook/Preview.png"
  },
  "sdk": {
    "name": "d3",
    "version": "host"
  },
  "runtime": {
    "dependencyMode": "host-plus-cache",
    "allowLifecycleScripts": false
  }
}

Rules

  • package.json keeps ecosystem identity: name, scripts, dependencies, dev dependencies, type, exports, and license.
  • appify.json should only describe Appify document concerns: kind, entry, canonical files, permissions, dependency mode, and migrations.
  • kind starts as webapp; SDK identity belongs in profile until a dedicated app graduates.
  • Do not use kind: "website" for npm-backed packages. That boundary is semantic, not cosmetic.
  • SDK packages imported by name should resolve to the app-owned SDK capsule when possible.
  • Package-local node_modules is an explicit sealed mode, not the default living-document mode.
  • dist/, .appify/cache/, and generated previews are derived unless the manifest marks them canonical.
