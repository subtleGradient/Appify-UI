JSON Canvas App Ecosystem

The JSON Canvas apps list is useful only if it separates durable storage targets from import/export bridges. AppifyHost should lean into apps whose source of truth is a local .canvas file and treat cloud or proprietary canvases as converters.

Apps Listed by JSON Canvas

App JSON Canvas support Current source of truth AppifyHost move Difficulty Golf read
Obsidian Storage, import, export. Vault-local .canvas files that reference Markdown, media, URLs, and other vault files. Keep building JSONCanvas.app as an Obsidian-compatible file editor. Add optional .canvasbundle import/export only when referenced assets need to travel. 2/5 for compatible .canvas; 5/5 for an Obsidian Canvas clone. Strong candidate. Low storage risk, high interoperability leverage, but do not imply Obsidian feature parity.
Kinopio Import, export. Kinopio space state, account/browser/cloud mediated; JSON Canvas is a lossy interchange path, not listed as storage. Build a converter or import/export helper, not Kinopio.app. A faithful local app would need its own .kinopio package and permission to bundle or patch the noncommercial client source. 4/5 for faithful persistence; 2/5 for one-way conversion. Mutate first. Good spatial UX evidence, but IndexedDB/cloud sync, lossy mapping, and licensing weaken document ownership.
Flowchart Fun Import, export. Text-to-flowchart source plus app/cloud state; project is open-source React and Cytoscape. Prototype a local text-first .flowchart or .flowchartfun document that exports JSON Canvas. Do not force every flowchart into raw .canvas storage. 3/5. Interesting, but mutate. Open source lowers build risk; the native object is the flowchart text, not JSON Canvas.
hi-canvas Import, export. Hosted web canvas/viewer state; public source and durable storage contract were not obvious from the live site. Treat as a compatibility probe for JSONCanvas.app. Avoid a dedicated wrapper unless source, offline mode, and file ownership are verified. 4/5. Defer. The app proves the format can be viewed online, but not that AppifyHost should wrap this product.
OrgPad Import, export. OrgPad cloud document; internal EDN .orgpage export can include uploaded files, while JSON Canvas is intentionally lossy. Build an OrgPad export importer only if EDN packages become common. JSON Canvas import should be best-effort and clearly lossy. 5/5 for faithful local persistence; 2/5 for explicit import/export conversion. Defer or converter-only. Cloud storage, undocumented EDN drift, rich cells, generated sizing, and uploaded assets make full persistence expensive.
Charkoal Storage, import, export. .canvas files in a codebase; Charkoal documents its format as a superset of JSON Canvas. Add independent Charkoal-compatible rendering and unknown-field preservation to JSONCanvas.app. Do not bundle the actual Charkoal UI without permission and a stable host contract. 2-3/5 for compatible files; 4/5 for the actual Charkoal UI. Strong candidate after JSONCanvas polish. Local files and Git sharing fit AppifyHost; proprietary packaging, VS Code/LSP coupling, and schema drift are the main burdens.

Persistence Pattern

Pattern Apps Native document stance First proof
Canonical local .canvas Obsidian, Charkoal The opened file can be the source of truth. Preserve unknown fields, write atomically, and detect stale saves. Open, edit, save, and round-trip real Obsidian and Charkoal samples without deleting app-specific data.
Portable asset package Obsidian-adjacent canvases, Charkoal code canvases .canvasbundle is justified only when local file nodes, images, or code snippets must travel together. Import a folder-backed canvas into a bundle, rewrite references, then export back to raw .canvas plus files.
Interchange bridge Kinopio, Flowchart Fun, hi-canvas, OrgPad JSON Canvas is an import/export format. AppifyHost should not pretend to control upstream persistence. Import/export fixtures and show exactly which fields are preserved, dropped, or approximated.
Native upstream package Flowchart Fun text, OrgPad EDN If a dedicated wrapper exists, it should own the upstream format first and export JSON Canvas second. Open the upstream file/package locally, save without cloud login, and produce a valid JSON Canvas export.

Build Ranking

Rank Build Why Thing Golf risk
1 JSONCanvas.app compatibility pass Covers Obsidian and Charkoal-compatible files with one local file editor. This is the lowest-burden path because the durable object is already .canvas. Safety low; burden moderate from unknown-field preservation and fixture coverage; betrayal risk is overclaiming app parity or bundling proprietary UI.
2 .canvasbundle import/export probe Solves the attachment portability gap without replacing raw .canvas. Safety moderate from path rewriting; burden contained if bundle rules stay narrow.
3 Flowchart Fun local text-document probe Open source and text-first. Good AppifyHost shape, but it is a flowchart app that exports JSON Canvas, not primarily a JSON Canvas editor. Chaos risk is inventing a second document model; keep source text canonical.
4 Kinopio and OrgPad importers Useful if users have exports. Bad fit for live persistence because the upstream products own cloud/account state. Betrayal risk is data loss through lossy conversion; make preservation reports explicit.
5 hi-canvas wrapper Not enough verified local-storage or source-code leverage to justify a dedicated wrapper. Burden and trust risk are high until source and offline persistence are clear.
