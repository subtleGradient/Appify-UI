DotOK Webform And Bundle Nouns

A DotOK document package should let a human double-click a .ok bundle, open a native DotOK.app document window, and iteratively design a specialized IDE by composing small durable nouns instead of concentrating the whole ontology into one overgrown tool.

Sequence: 000 webform and bundle nouns

Raw Capture

Think about a .appdev bundle for the DotOK project: double-click a .ok bundle and it opens in a new DotOK.app document window.

Inside a DotOK bundle are many things. Each thing has zero or more sub-things, which are also things. Each thing has many aspects. Each aspect has one way in which it can be good, healthy, and stable: its homeostatic balance point, the needle hole to thread.

Each aspect may have many bad failure modes: too high, too low, too far left or right, the wrong material, incomplete passage, and so on. Assessments are point-in-time reports about how reality is now, including snapshots of the things, aspects, values, and context true at that time.

There also needs to be a way to describe a future snapshot that does not yet exist. This is not a plan yet. It is a jumping-off slice that focuses some subset of things and aspects. Goals and work orders flow from there. Work orders may unblock one or many goals or other work orders.

Follow the REST philosophy in a different way: decompose complexity into nouns. Each noun is either a custom filename for a standard format, like .canvas for JSON Canvas, or a bundle folder that encapsulates related stuff, like web components and Shadow DOM.

DotOK.app can handle many bundles and formats by exposing a different Bun server for each kind. A dotok:/// scheme can let nouns naturally open in the correct DotOK view. This avoids smuggling complexity everywhere and reduces cognitive overwhelm by composing a few simple patterns repeatedly.

The even more basic missing primitive is a web-tech version of a fillable PDF form: .webform documents that open in WebFormer.app. The initial sketch used a bundle with form.html plus data.jsonp; the stronger direction is a single HTML file whose visible controls are the saved state.

Fancy form inputs, JSON editors, Markdown controls, and related components should live in the sibling /Users/tom/Developer/web-native/ project. WebFormer can inject those affordances into the served response; the source .webform stays clean, printable, parseable HTML unless the author intentionally chooses richer markup.

First Dependency: .webform

Before DotOK can become a specialized IDE for homeostatic reality modeling, it needs a boring, portable, inspectable way to collect and preserve structured human input. Treat .webform as the fillable-PDF primitive for the whole system.

Native When Available

WebFormer.app opens a single .webform HTML file, serves a rewritten response with injected controls and styles, validates every save, then rewrites the source HTML with Bun HTMLRewriter.

Plain Web Otherwise

Without WebFormer.app, the user opens the .webform file as ordinary HTML. The current answers are already in value, checked, selected, and textarea content, so the document remains inspectable and printable.

Components Live Elsewhere

Rich controls belong in web-native: JSON editors, Markdown inputs, radios, toggles, tabs, progress, chat transcript renderers, and other reusable custom elements. DotOK should consume them through pinned CDN imports, not absorb their source.

PDF-Like Mental Model

The user should understand a .webform as "the form plus my filled answers." It is not a database, not a web app project, and not a task engine. That constraint is what makes it dependable.

Proposed .webform File

client-intake.webform

A .webform is an HTML document with a document extension, not a package. The durable state is the HTML itself. WebFormer may create exports beside it, but exports are derived artifacts, not part of the canonical format.

Primary Job

The core job is agent-to-agent decision making, where an agent may be human or AI. A good .webform is therefore a semantic artifact first: readable headings, explicit choices, visible current values, inspectable tradeoffs, and no hidden state that a reviewer cannot see, edit, quote, or diff.

Convention-First Webform Contract

Surface Role Rule
*.webform Source document A single HTML file. No required bundle, manifest, sidecar JSON, embedded app script, embedded CSS, or build step.
Native controls Durable state Saved answers live in ordinary HTML: value, checked, selected, and textarea text. If state exists, it should be visible or clearly represented in the document.
Names and ids Optional handles Authors add id or name only when useful for human or agent references. WebFormer can inject ephemeral names into the served response when the source omits them.
Injected runtime Disposable affordance WebFormer injects scripts, styles, save buttons, diagnostics, field addresses, source hashes, and richer controls into the HTTP response. Those affordances are not written into the source file unless explicitly requested.
Exports Derived artifacts PDF, filled static HTML, snapshots, and attachments can be produced beside the file. They are outputs, never hidden source-of-truth state.

Sidecar Correction

data.jsonp is rejected as the canonical webform data file. Real JSONP is data wrapped in a callback, such as callback({ ... }), not an arbitrary generated hydrator script. Pure data.json is safer, but a sidecar still creates invisible state and synchronization risk. The stronger v0 rule is: no sidecar data at all. The source HTML is the saved state.

Standards Reality

  • Modern ESM JSON import attributes are useful, but they do not beat a single HTML source file for this job. They need module semantics and correct MIME behavior, and browsers do not natively import JSON5.
  • Classic HTML form controls already have durable state locations: attributes for many controls, text content for textarea, and selected state for options.
  • FormData remains the right transport shape for WebFormer saves, but not the durable storage shape. WebFormer translates request data back into HTML.
  • Form-associated custom elements and Declarative Shadow DOM are useful for rich controls, but they are not v0 primitives. Custom controls should mirror to native controls or participate in form submission cleanly.

WebFormer Rewrite Model

WebFormer owns runtime complexity. On GET, it reads the source HTML and uses Bun HTMLRewriter to inject a save UI, diagnostics, source hash, generated control names, and optional web-native components into the response. On POST, it verifies the source hash and field map, rewrites the source HTML, reparses it, and refuses to save if the round trip is not lossless. 

GET client-intake.webform
  read source HTML
  parse/rewrite response with HTMLRewriter
  inject generated names only into the served response
  inject diagnostics and save controls outside source forms

POST client-intake.webform
  verify source hash
  reject unknown submitted fields
  reject missing expected fields unless intentionally disabled
  rewrite source control state:
    input value / checked
    textarea text
    option selected
  reparse and verify visible state round-trips
  write source or return a blocking diagnostic

Zero Silent Errors

  • Unknown submitted field: hard error. WebFormer never drops it.
  • Source changed since page load: hard conflict. WebFormer never writes over a stale hash.
  • Generated field address no longer resolves to exactly one source control: hard error.
  • Duplicate explicit ids used as stable handles: hard error.
  • Persistent hidden state: hard warning or hard error. The default webform must be human- and agent-visible.
  • Multiple forms: allowed. WebFormer treats each form as a separate decision surface and injects save controls with form-specific source hashes.
  • Controls associated by the native form attribute: supported only when the association is unambiguous; otherwise WebFormer blocks the save with a visible report.
  • Custom elements: supported only if they mirror to native controls or are form-associated custom elements with inspectable values.
  • Round-trip mismatch after rewrite: hard error with before/after diagnostics.

DotOK Core Ontology

Noun Meaning Likely Representation
thing Any object of care. A thing may contain sub-things, which are also things. things/*.thing/ bundle, thing.json5, optional child refs.
aspect A dimension along which a thing can be healthy, stable, drifting, broken, ambiguous, or unknown. aspects/*.aspect.json5 or nested aspects/*.aspect/ when it needs examples, forms, and evidence.
balance point The one homeostatic way the aspect is good: the needle hole to thread. Named assertion, evaluator, or form section owned by the aspect.
failure mode One recognizable way to miss the balance point: too much, too little, wrong direction, wrong material, incomplete connection, stale evidence. failure-modes/*.mode.json5, examples, detection prompts, tests, or calibration notes.
assessment A point-in-time reality snapshot. It captures the then-current thing graph, aspect definitions, observed values, evidence, and judgment. assessments/YYYYMMDD-HHMM.assessment/ with frozen copies or content-addressed refs.
future snapshot A desired not-yet-real state slice. It is not a plan and does not own tasks. It frames a tractable slice of reality to orient goals. futures/*.future/ bundle, often backed by .webform, .canvas, and projection notes.
goal A high-level desired outcome derived from a future snapshot or assessment gap. goals/*.goal.json5 plus links to aspects, things, futures, and evidence.
work order An executable unit that can unblock goals or other work orders. This is the DotOK task edge, not the whole world model. work-orders/*.work.json5 or compatibility with existing .tasks/*.md YAML frontmatter.

Important Naming Tension

The current DotOK protocol already uses <FOCUS_ROOT>/.ok/ as the hidden evergreen ideal-state directory. A Finder document named My Project.ok is different, but the conceptual overlap is sharp enough that the first implementation needs an explicit bridge.

  • Option A: .ok becomes the visible Finder package for new DotOK documents, while legacy project roots keep hidden .ok/.
  • Option B: .ok stays the hidden protocol directory and the Finder package uses .dotok or .okbundle.
  • Option C: DotOK.app opens both: visible *.ok/ packages as documents and regular project folders by discovering their hidden .ok/ and .tasks/ peers.

REST-Shaped Nouns, Not Tool Soup

The design move is to make every conceptual pressure turn into a noun with a small handler. DotOK.app is then an orchestrator over nouns, not a giant bespoke verb machine.

Kind Handler Default View Example URI
.webform webform-server Form fill, saved response, print/export dotok:///open?path=things/product.thing/aspects/reliability.aspect/balance.webform
.canvas canvas-server Graph, focus slice, dependency map dotok:///view/canvas/home.ok.canvas#node=aspect-reliability
.thing thing-server Thing overview, child things, aspect status dotok:///thing/product
.assessment assessment-server Frozen reality report and evidence dotok:///assessment/20260522-1430
.future future-server Not-yet-real target slice dotok:///future/launch-ready
.goal goal-server Outcome, current blockers, linked slices dotok:///goal/reduce-onboarding-drift
.work work-server Execution order, blockers, unblocked dependents dotok:///work/add-observable-startup-path

Bun Server Per Kind

  1. DotOK.app receives a document URL from Launch Services or a dotok:/// URI.
  2. It resolves the target noun and starts or reuses the matching app-local Bun server.
  3. The server exposes a narrow local API for that noun kind: read source, render default view, validate edits, save edits, export, and link to neighbors.
  4. The document window is just a local web surface pointed at the current noun server.
  5. Cross-noun navigation happens through links, not shared global state: dotok:///thing/product, dotok:///assessment/..., dotok:///work/....

Likely Build Sequence

  1. Build the smallest single-file .webform fixture with one native HTML form, visible saved values, no embedded scripts, no embedded styles, no sidecars, and no required control names.
  2. Prototype WebFormer.app as an AppifyHost app that registers .webform, serves the file through a local Bun server, injects runtime affordances with HTMLRewriter, and saves by rewriting the HTML source.
  3. Write down the fixed conventions for field addressing, source hashes, multi-form handling, zero-silent-error diagnostics, and exports. Add manifests or sidecars only after a real fixture proves HTML-as-state cannot carry a needed declaration.
  4. Create a toy needle-threading.ok bundle with one thing, one aspect, one balance point, several failure modes, one assessment, one future, one goal, and two work orders.
  5. Prototype DotOK.app opening the .ok bundle and dispatching to kind-specific local servers.
  6. Add dotok:/// routing after file opening works, so links become integration glue instead of the primary persistence model.
  7. Only then start designing the "unlike anything seen before" IDE views, using the noun contracts as load-bearing constraints.

Uncertainties And Decisions

  • How stable generated field addresses should be across edits when authors omit id and name.
  • Whether hidden controls are ever allowed as persistent state. The v0 bias is no, because invisible state breaks agent-to-agent inspectability.
  • How custom elements participate without sidecar data. Form-associated custom elements or hidden native input mirrors may be the right boundary.
  • How file inputs and attachments round-trip, since browsers intentionally do not let scripts repopulate <input type="file">.
  • How to prevent injected web-native components from turning a portable form into a supply-chain gamble. Pinning to commit SHAs seems mandatory for serious forms.
  • Whether complex forms eventually need an optional manifest, and what concrete pressure justifies crossing that line.
  • Whether WebFormer should preserve author formatting exactly or normalize rewritten HTML. Exact preservation is nicer; deterministic normalization may be safer.
  • How much DotOK snapshot content is copied versus content-addressed. Assessments need to be stable even when live thing definitions evolve.
  • Whether failure modes are global reusable patterns, aspect-local children, or both.
  • Whether future snapshots should be immutable once used to generate goals, or whether they are editable design targets with versioned revisions.
  • How work orders map to the existing DotOK .tasks/*.md YAML protocol without creating two task systems.
  • How dotok:/// handles moved bundles, multiple open documents, URL encoding, permissions, and cross-document links.
  • How to keep the IDE cognitively quiet when the underlying graph becomes enormous. The noun system lowers complexity pressure, but it does not remove view-design responsibility.
