Web Components Native

A ludicrously simple native app model: render real SwiftUI components, run JavaScript in a hidden WKWebView, describe the native tree with web components, and connect the two with postMessage. Let SwiftUI do what SwiftUI is good at. Let web technology do what web technology is good at.

Origin

I was one of the original founding engineers on React Native. The project eventually went in a different direction than the one I originally proposed. This is a return to that earlier impulse, with the benefit of modern SwiftUI, modern WebKit, and modern web components.

React Native proved that JavaScript can productively drive native applications, but it drifted toward owning layout, rendering, reconciliation, and platform abstraction as if it were a game engine. That is a valid direction for a company that wants to minimize platform dependency. It is not the only direction.

This returns to the earlier root: JavaScript as the application brain, native frameworks as the application body. I do not mind being controlled by Apple when building Apple software. The operating system is the product surface. Use it.

Principles

  • No React runtime.
  • No Yoga layout engine.
  • No custom renderer pretending to be the platform.
  • No attempt to make iOS, macOS, Android, and web look identical.
  • Use web components as the JavaScript-facing declaration ABI.
  • Use SwiftUI as the rendering, layout, animation, accessibility, focus, menu, and platform integration layer.
  • Use a hidden web view for JavaScript modules, async work, routing, storage, fetch, WebSocket, workers where available, and the broader web ecosystem.
  • Keep the bridge small enough that it can be understood in one sitting.

JavaScript Surface

The application author writes plain HTML-shaped declarations. Custom elements do not render DOM pixels. They describe native SwiftUI views and keep JavaScript ergonomics: attributes, properties, slots, events, and composition. 

<script type="module">
  import "web-components-native/swiftui/define.js"

  document.body.innerHTML = `
    <native-app>
      <native-window title="Settings">
        <native-vstack spacing="16" padding="20">
          <native-text font="title2">Account</native-text>
          <native-text-field
            id="email"
            label="Email"
            value="tom@example.com">
          </native-text-field>
          <native-toggle id="notifications" checked>
            Notifications
          </native-toggle>
          <native-button id="save" role="primary">
            Save
          </native-button>
        </native-vstack>
      </native-window>
    </native-app>
  `

  document.getElementById("save").addEventListener("native:press", () => {
    console.log("save", document.getElementById("email").value)
  })
</script>

Bridge Protocol

The hidden web view observes the custom-element tree, diffs it, and sends small patches to Swift. Swift keeps stable native identities and applies the patches to an observable model consumed by SwiftUI. 

{
  "type": "wcn.patch",
  "transaction": 42,
  "root": "document",
  "ops": [
    {
      "op": "mount",
      "id": "save",
      "parent": "settings-stack",
      "component": "Button",
      "props": { "role": "primary", "title": "Save" }
    },
    {
      "op": "set",
      "id": "notifications",
      "props": { "checked": true }
    }
  ]
}
{
  "type": "wcn.event",
  "id": "save",
  "event": "press",
  "payload": {}
}

Native Responsibilities

  • Render SwiftUI views from a decoded component tree.
  • Own native view identity, focus, accessibility, selection, keyboard behavior, sheets, menus, and window chrome.
  • Use platform-native controls before inventing custom controls.
  • Send user events, measurements, permission results, and lifecycle notifications back to JavaScript.
  • Expose capabilities explicitly: files, notifications, camera, pasteboard, shell, app menus, windows, and document packages.
  • Allow real visible WKWebView islands when the app actually needs web rendering.

Web Responsibilities

  • Own application state that benefits from JavaScript: data loading, syncing, routing, validation, local persistence, AI clients, and protocol glue.
  • Use DOM custom elements as the authoring model and bridge observation surface.
  • Use browser-native APIs where they are better than rebuilding infrastructure in Swift.
  • Keep component definitions as plain ESM, aligned with web-native: no bundler requirement, no framework runtime requirement, no TypeScript source requirement.
  • Make demos openable as plain HTML whenever possible.

React Native Contrast

  • React Native: JavaScript describes a React tree. A custom renderer maps that tree into host views.
  • Web Components Native: JavaScript describes a DOM tree. Custom elements map that tree into SwiftUI views.
  • React Native: cross-platform sameness is a core pressure.
  • Web Components Native: platform fit is the core pressure.
  • React Native: the framework often competes with the platform.
  • Web Components Native: the framework chooses the platform and then gets out of the way.

Prototype Path

  1. Start macOS-only, inside the existing Appify UI SwiftUI/WebKit lineage.
  2. Create a hidden WKWebView runtime with one registered message handler.
  3. Load a local HTML document that imports web-components-native/swiftui/define.js.
  4. Implement a custom-element observer that serializes the declaration tree into patches.
  5. Decode patches into an ObservableObject Swift model.
  6. Render a small set of SwiftUI components from that model.
  7. Return button, toggle, text-field, lifecycle, and measurement events to JavaScript.
  8. Ship one tiny settings-window demo and one document-style app demo.

Non-goals

  • Do not clone CSS layout in Swift.
  • Do not chase pixel parity across platforms.
  • Do not replace SwiftUI with a canvas.
  • Do not hide platform conventions behind generic widgets when native controls exist.
  • Do not require npm, bundlers, JSX, React, or a compile step for the simple path.

Open Questions

  • How much SwiftUI layout should be exposed directly versus wrapped in simpler custom elements?
  • How should native-only transient state be preserved across JavaScript patches?
  • Should the declaration tree be DOM-first, JSON-first, or dual?
  • How should backpressure work when JavaScript sends patches faster than SwiftUI should animate?
  • What is the smallest capability security model that still feels native and useful?
  • How much of this should become a standalone framework versus staying an Appify UI convention?
