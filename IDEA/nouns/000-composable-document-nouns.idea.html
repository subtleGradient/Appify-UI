Composable Document Nouns

Use fractal decomposition: every complexity pocket becomes a small noun with a file shape, a handler, links to neighbors, and fail-closed rules. Big systems stay understandable by composing those nouns instead of centralizing all behavior.

Depends on: single-file webform / Used by: DotOK stack

Fractal Rule

  1. If a concept can stand alone, give it a standalone noun.
  2. If a noun is just one document, prefer a custom filename extension over a bundle.
  3. If a noun needs related artifacts, make it a bundle folder with a tiny source-of-truth file.
  4. If a noun needs runtime behavior, give it a kind-specific local server.
  5. If nouns need to compose, link them through explicit refs or app URIs, not shared hidden state.

Document Shapes

Shape Use When Examples
Single file The artifact is fully understandable as one standard-format document. .webform, .canvas, .idea.html, .ok.md
Bundle folder The noun owns multiple files that must move together. .thing/, .assessment/, .future/, .appdev/, .ok/
Generated export The file is useful for sharing or printing but not the source of truth. latest.pdf, static filled HTML, thumbnails, Quick Look previews

Kind Servers

Each noun kind should have a small local handler that owns exactly the behaviors needed for that kind. This keeps the host app from becoming a large object with random verbs.

Kind Handler Default View
.webform webform-server Visible form state, save diagnostics, exports
.canvas canvas-server Graph, focus slice, dependency map
.thing/ thing-server Thing overview, children, aspects, evidence links
.assessment/ assessment-server Frozen reality snapshot and evidence
.future/ future-server Not-yet-real target slice
.work work-server Execution state, blockers, unblocked dependents

URI Glue

App URIs should open nouns in the right view without forcing every file to know every app detail. A URI points to a noun and optional view state; the kind server resolves what that means. 

dotok:///open?path=things/product.thing/aspects/reliability.aspect/balance.webform
dotok:///view/canvas/home.ok.canvas#node=aspect-reliability
dotok:///thing/product
dotok:///assessment/20260522-1430
dotok:///work/add-observable-startup-path

Pit Of Success

  • Prefer visible source of truth over invisible sidecars.
  • Prefer convention over manifests until a real fixture proves convention cannot carry the case.
  • Prefer native standard formats over custom JSON when humans or agents need to inspect the artifact.
  • Prefer one handler per noun kind over one generic manager with many special cases.
  • Fail closed on unknown fields, stale source hashes, ambiguous refs, and round-trip mismatches.

Open Decisions

  • How much naming convention belongs in the generic layer versus each app.
  • Whether bundle manifests should share one minimal shape or remain kind-specific.
  • How app URI permissions work across multiple open documents.
  • How generated exports declare their source without becoming source-of-truth themselves.
