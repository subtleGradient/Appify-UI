Single File WebForm Standard

A .webform should be a single HTML file whose visible form controls are the saved state. WebFormer can add runtime affordances while serving it, but the durable artifact stays plain, inspectable, printable HTML.

Builds into: composable document nouns / DotOK stack

Primary Job

The main job is agent-to-agent decision making, where an agent may be human or AI. A useful webform is a semantic artifact: headings, labels, fieldsets, choices, current values, and tradeoffs are visible in the file itself. No hidden sidecar state is required to understand what the form currently says.

V0 Contract

Surface Rule
*.webform One HTML file. No required bundle, manifest, sidecar JSON, embedded app script, embedded CSS, or build step.
Durable state Saved answers live in native HTML: value, checked, selected, textarea text, and vetted contenteditable HTML.
Source style Lean into HTML5's forgiving syntax. Omit optional html, head, and body tags when they add noise. Use unquoted attribute values when valid and readable. Prefer flat, low-indent source.
Names and ids Optional. Add id or name only when useful for a human or agent reference. WebFormer can inject temporary names while serving.
Runtime affordances Injected by WebFormer into the served response with Bun HTMLRewriter. They are not part of the source file unless explicitly saved as source.

Web-Native Editing

WebFormer should support native browser editing affordances when they keep the source file inspectable. The important rule is not "no rich UI"; the rule is that the rich UI must resolve back to visible HTML that another human or agent can read without WebFormer.

Affordance Native Surface Save Rule
Rich text contenteditable, selection, Input Events Save a sanitized, allowed subset of HTML inside the editable element. Never hide a parallel rich-text document model.
Paste Clipboard/DataTransfer payloads surfaced through input and paste events Prefer semantic HTML. If pasted data cannot be represented visibly and safely, block or ask instead of dropping data.
Drag reorder HTML Drag and Drop events plus DataTransfer Persist the reordered DOM source with minimal patches and round-trip verification.
File drop HTML Drag and Drop file items, file inputs Embed a visible link, figure, or native control state in the HTML. Never create invisible sidecar state.

Diff-Friendly Saves

WebFormer should behave like a careful structured editor, not a formatter. When a user or agent edits a value, the save path should modify the smallest source span that represents that value and leave unrelated whitespace, optional tag choices, attribute order, quote style, indentation, comments, and surrounding markup untouched. Clean git diffs are part of the format's trust model.

  • Changing a text input should update only that input's value attribute.
  • Changing a textarea should update only its text content.
  • Changing a contenteditable region should update only that region's vetted inner HTML.
  • Changing a checkbox, radio, or option should update only the relevant checked or selected state.
  • Drag-and-drop reorder operations should move the smallest stable element range rather than reserializing the whole document.
  • Do not add optional html, head, or body tags just because the parser exposes them in the DOM.
  • Do not quote or unquote existing attributes unless the edited value requires it.
  • Normalization is an explicit repair action, not the default save behavior.
  • If a minimal patch cannot be proven safe, WebFormer should block with a diagnostic instead of silently reformatting the document.

Zero Silent Errors

  • Unknown submitted field: hard error.
  • Stale source hash: hard conflict.
  • Generated field address resolves to zero or multiple source controls: hard error.
  • Duplicate explicit ids used as stable handles: hard error.
  • Persistent hidden state: hard warning or hard error.
  • Multiple forms: allowed, but WebFormer treats each as a separate decision surface.
  • Native form attribute associations: supported only when unambiguous.
  • Custom elements: supported when they mirror to native controls or are form-associated custom elements with inspectable values.
  • Rich text paste/drop includes disallowed markup or hidden payloads: hard diagnostic with a visible cleanup choice.
  • Dropped file cannot be represented as visible HTML or native control state: hard diagnostic.
  • Round-trip mismatch after rewrite: hard error with before/after diagnostics.

Rejected V0 Shapes

  • data.jsonp: executable data and a historic cross-domain security hole.
  • data.json sidecar: safer than JSONP, but still creates invisible split state and synchronization risk.
  • Mandatory manifests: premature configuration surface.
  • Embedded app code in every form: turns a document into a tiny app project.
  • Required field names: burdens authors before the name is useful.
  • Opaque rich-text blobs: useful editing UI, bad agent artifact.

Open Decisions

  • How stable generated field addresses should remain across edits when authors omit ids and names.
  • Whether hidden controls are ever allowed as persistent state. V0 bias: no.
  • What exact HTML subset is allowed for contenteditable saves.
  • How file inputs and dropped files round-trip, since browsers intentionally do not let scripts repopulate <input type="file">.
  • How much source-location metadata WebFormer needs to preserve minimal patches across malformed-but-browser-recoverable HTML.
