SDKs Are Not Schemas

JSON Canvas works because the file format is intentionally small and fixed. These SDKs are closer to programmable surfaces: each can render many possible products, so the document boundary has to include code, data, and runtime policy.

Read order: index / webapp vs website / jobs / Webapp first / layers / package contract / dependencies / SDK profiles / golf / probes

The Contrast

Format What opens What the app must know Failure mode
JSON Canvas A stable JSON graph with nodes, edges, positions, and attachments. The spec. Editor quality can be weak, but the document boundary is clear.
Cytoscape.js Graph elements, style, layout, app data, and sometimes custom UI code. Graph JSON plus conventions for assets, layout, and custom panels. A generic graph viewer is useful but not the same as the user's app.
Rete Nodes and connections that may be plain JSON or JS class instances. A serializer and a node registry. Saving arbitrary editor objects can silently lose behavior.
React Flow Nodes, edges, viewport, custom node renderers, and app-level semantics. React components and a flow JSON convention. The flow renders, but custom nodes have no meaning.
tldraw/TLCanvas Store snapshots, assets, migrations, session, and custom shapes. The document/session split and shape migration boundaries. IndexedDB becomes the real document by accident.
D3 A visualization program over data, not a D3-native document file. Normal web package entry, source, data paths, and dependency resolution. Inventing a fake universal .d3 JSON format collapses the ceiling.

Tempting Wrong Moves

  • Define one new universal graph schema for all five. That lowers the ceiling and loses the ecosystem.
  • Ship every package with node_modules. That makes documents huge, platform-shaped, and noisy in git.
  • Auto-run package scripts on open. That makes double-clicking a document feel like executing untrusted software.
  • Use browser storage as the source of truth. That breaks Finder copy, Time Machine, git, rsync, and inspection.
  • Blur .webapp into .web or .website. Static website bundles should never imply npm, package-manager installs, or build steps.
  • Build five bespoke hosts before a generic .webapp host. That duplicates the same package inspection, dependency cache, and save bridge problems before the common layer is known.

The Better Shape

  • Make the package itself normal enough that web developers recognize it: package.json, index.html, src/, data/, public/, and relative assets.
  • Prove that normal shape first in Webapp.app with .webapp examples for each SDK.
  • Keep .web and .website for static web-native bundles that already run as authored browser files.
  • Let AppifyHost provide the document shell, safe local server, path environment, window identity, and navigation restrictions it already knows how to provide.
  • Let each app provide an SDK capsule: pinned SDK packages, host-owned adapters, starter templates, and a save bridge.
  • Add appify.json only when the package needs more specificity than ecosystem-normal conventions provide.
