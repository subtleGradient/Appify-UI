Where node_modules Lives

The package should not carry the SDK runtime by default. The app owns the SDK capsule, the package owns source and lockfiles, and extra dependencies live in a cache unless the package deliberately seals them.

Read order: index / problem / jobs / layers / package contract / SDK profiles / golf / probes

Recommended Policy

  • The app bundle owns d3, cytoscape, rete, @xyflow/react, tldraw, React, build adapters, and the local runner versions it supports.
  • The package owns package.json, lockfiles, source, data, assets, and any explicit compatibility declarations.
  • A per-user or per-app content-addressed cache owns downloaded extra dependencies and transformed output, keyed by package path plus lockfile hash.
  • The opened package is never mutated by dependency install unless the user explicitly chooses a write action.
  • Lifecycle scripts are off by default. If enabled, they become a visible package capability.

Strategies

Strategy Good Bad Verdict
Always bundle node_modules inside every document Offline and self-contained. Huge, noisy, platform-sensitive, stale, and bad in git. Only for sealed/archive mode.
Only use app-owned SDK dependencies Fast, safe, small, reproducible. Ceiling is too low for real packages with extra libraries. Good default floor, not enough alone.
App-owned SDK plus lockfile-keyed cache Small documents, normal dependency declarations, reproducible enough. Needs cache management and clear missing-dependency UX. Default living-document mode.
Auto-install into package on open Feels magical once. Surprise writes, surprise network, surprise scripts, trust break. Do not do this.

What Is A Normal D3 Package?

For this purpose, a normal D3 package is a small web project, not a published npm library and not a special D3 file format. It has a package.json, an HTML or declared module entry, ESM source that imports d3, and data/assets addressed by relative paths.

Resolution Order

  1. Use package-local sealed dependencies if appify.json asks for sealed mode.
  2. Resolve known SDK imports from the app-owned SDK capsule.
  3. Resolve declared extra dependencies from the lockfile-keyed cache.
  4. If extra dependencies are missing, show a dependency resolution page with exact package names and cache target.
  5. Never run package lifecycle scripts unless the package capability and user action both allow it.
