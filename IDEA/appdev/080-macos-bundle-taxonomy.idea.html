macOS Bundle Taxonomy

The previous note collected bundle conventions as a pile. This note sorts them by category: generic packages, modern bundles, app bundles, document packages, code-bearing nested bundles, metadata bundles, helper-process bundles, and niche platform bundles.

Sequence: 000 capture / 060 hello world / 070 conventions pile / Quick Look spec

Core Distinction

Package

A directory presented by Finder as one file. Package-ness is a user interface behavior. Apps, plug-ins, frameworks, document packages, and installer packages can all be packages.

Bundle

A directory with a standardized hierarchy for code and resources. Bundle-ness is a developer/runtime contract. Many bundles are also packages, but document packages are not necessarily bundles.

Appify reading

.appdev, .tlcanvas, and .hi should be document packages first. .app should be the code-bearing bundle that registers, opens, previews, indexes, and explains those packages.

Category Map

Category Canonical examples Applies to Free behavior Compatibility depth Appify stance
Generic packages Any directory Finder treats as one file: .app, .bundle, .framework, app-claimed document packages All package directories, whether code-bearing or not Finder opacity, drag-install ergonomics, Show Package Contents, accidental-edit protection Deep macOS lineage. Apple documents package recognition by known extension, claimed extension, or package bit. Make .appdev, .tlcanvas, and .hi packages, but do not make them full bundles unless runtime lookup needs it.
Modern macOS bundles Contents/Info.plist, Contents/MacOS, Contents/Resources, Contents/Frameworks, Contents/PlugIns, Contents/SharedSupport App bundles and loadable bundles; frameworks use a different versioned shape Bundle identity, resource lookup, localization, executable lookup, code signing shape, private support files Modern-style macOS bundle. Apple explicitly says Contents separates modern bundles from document and legacy bundle types. Generated .app bundles should be boringly conventional here. Put runtime-required files in Resources; optional hackable examples in SharedSupport.
Application bundles MyApp.app/Contents/Info.plist, main executable, app icon, document declarations, URL declarations, Services, Help Book, privacy manifest Only launchable apps Double-click launch, Dock/Finder identity, Open With, document ownership, app-local resources, native Help menu, deep-link entry points Old and central. Some keys are legacy, some are modern. The app bundle is the main integration point for AppifyHost. AppifyHost templates should aggressively own this category. The .app is the distributable, hackable, self-contained product.
Document packages Report.pages, Canvas.tlcanvas, Hello.hi, future App.appdev Directories registered by an app as document types and presented as one file Complex internal files with a simple user-facing document object; package-local Quick Look preview files; standard filesystem access Durable macOS pattern. Apple does not prescribe internal document package structure and says NSFileWrapper is the Cocoa-preferred write path. Keep package contents legible: QuickLook/Preview.*, JSON5 manifests when needed, assets, notes, and source truth. Avoid nested Contents unless the document is intentionally also a bundle.
Type and Launch Services declarations UTExportedTypeDeclarations, UTImportedTypeDeclarations, CFBundleDocumentTypes, LSItemContentTypes, LSTypeIsPackage, LSHandlerRank Mostly app bundles, describing documents they create/open Kind identity, file association, package treatment, document icons, Open With ranking, pasteboard/content-type interop Mixed. Four-character OS types are legacy; UTIs/UTTypes are the modern layer. LSItemContentTypes supersedes older extension/MIME/OSType declarations on macOS 10.5 and later. Mandatory for serious AppifyHost apps. Prefer explicit UTTypes and package conformance. Include legacy fields only if a concrete compatibility target needs them.
Framework bundles Thing.framework/Versions/A, Versions/Current, Headers, Resources Shared libraries with associated resources; public frameworks and private app-embedded frameworks Resource-bearing dynamic libraries, versioned layout, stable link/import object Historically deep. Apple says framework structure predates macOS and supports multiple framework versions for older applications. Defer. Use private frameworks only when AppifyHost native code is large enough to justify rpaths, signing, and binary lifecycle.
Loadable bundles and plug-ins .bundle, .plugin, app-owned modules under Contents/PlugIns Code loaded into a host application's process Lazy feature loading, host-defined extension points, app-local modularity Old macOS feature. Apple describes loadable bundles as app-bundle-like directories with no normal main entry point. Only for intentional Appify plug-in APIs. Do not use plug-ins to solve plain file organization.
App extensions Finder Sync, Finder Action/Action extension, Share extension, Quick Look Preview extension under Contents/PlugIns Containing app plus system host processes System-hosted UI and actions in Finder, Share sheets, Quick Look, and other extension points Modern compared with classic bundles. Requires extension Info.plist keys, entitlements, host lifecycle discipline, and usually a separate target. Prototype only after passive package conventions fail. Finder Sync in particular is not a general Finder modification toy.
Quick Look conventions Package-local QuickLook/Preview.* and Thumbnail.*; older generator plug-ins; modern preview extensions Document packages first; app extensions only for dynamic formats that cannot store previews Finder previews and thumbnails, often without custom extension code Stable enough to standardize for Appify. Static package previews are the lowest-cost path. Already adopted: .tlcanvas/QuickLook/Preview.png and .hi/QuickLook/Preview.html.
Spotlight metadata bundles Contents/Library/Spotlight/*.mdimporter Apps that own custom document types Custom document metadata becomes searchable in Spotlight and Finder search Classic Mac OS X era. Importers are plug-in bundles and must not depend on the creating app running. Strong candidate for .appdev once metadata vocabulary stabilizes: app name, document types, notes, task titles, bundled app summary.
Helper process bundles Contents/XPCServices/*.xpc, login helper apps managed through SMAppService Apps that need isolation, crash containment, privilege separation, or background/login behavior On-demand helpers, launchd management, separate sandbox, crash isolation, user-approved background presence Modern. XPC Services documentation distinguishes low-level Mac OS X 10.7 compatibility from higher-level 10.8 APIs. Defer until a renderer, parser, indexer, or build worker is dangerous enough to deserve a process boundary.
Native command surfaces CFBundleURLTypes, NSServices, App Intents, App Shortcuts Mostly app bundles; App Intents are Swift code plus system metadata Deep links, selected-text/file actions, Shortcuts and Spotlight action discovery Mixed. URL types and Services are older plist surfaces; App Intents are modern. URL schemes and Services are good near-term experiments. App Intents should wait until Appify commands are stable and human-language-friendly.
Help and localized resource bundles *.lproj, InfoPlist.strings, Localizable.strings, Help Books via CFBundleHelpBookFolder and CFBundleHelpBookName All modern bundles for localization; app bundles for Help menu integration Localized names/resources, native Help menu, Help Viewer lookup Very old and very durable. Human-readable English.lproj-style names are still supported for backward compatibility, though ISO names are preferred. Use .lproj if we localize. Consider Help Books for self-documenting generated apps, but plain HTML in SharedSupport may be enough first.
Niche platform bundles Preference panes, Quartz Composer plug-ins, WebKit plug-ins, legacy Quick Look generators, legacy Dashboard widgets, installer packages Specific host apps or system features Host-specific extension and distribution behaviors Varies wildly. Some are deprecated or historical, some remain useful in narrow contexts. Learn the shape, but do not copy the commitment. Promote only when an Appify app truly wants that host surface.

Compatibility Strata

Stratum Examples Reliability note Appify implication
Deep lineage Directory-as-object packages, localized .lproj resources, framework versioning, four-character package/type codes Some exact history reaches back through NeXTSTEP and classic Mac OS families, but modern behavior should be judged by current macOS docs and tests. Copy durable ideas, not nostalgia. The durable idea is "a directory can be a coherent object".
Universal modern bundle substrate Info.plist, bundle identifier, executable/resource lookup, Resources, .lproj, CFBundleVersion Applies across modern bundle types, with structural variation for apps, frameworks, and plug-ins. Generated code-bearing bundles should look ordinary to NSBundle, Launch Services, code signing, and humans.
App-only registration layer Document types, URL schemes, Services, app icon/name, Help Book, app category, privacy manifest The app is where macOS expects ownership declarations and many user-facing integrations. Put declarations in the generated AppifyHost app, not inside every document package.
Document-package affordance layer LSTypeIsPackage, package UTType conformance, QuickLook/Preview.*, NSFileWrapper The owning app registers the type; the package stores the content. Apple does not prescribe internal layout. Appify packages can be opinionated but should stay readable. QuickLook is the blessed macOS exception to platform-neutral internal names.
System-hosted extension layer Finder Sync, Action/Share extensions, Quick Look preview extensions, Spotlight importers These are powerful but introduce separate lifecycle, signing, entitlements, and host-process constraints. Use after the passive file/plist convention is not enough.
Process-isolation layer XPC services, login helpers, background agents These change trust, state, and failure semantics. Reserve for dangerous parsing/rendering, indexing, build workers, and user-visible background promises.

Proposed Appify Rules

  1. .app is the self-contained runtime and source distribution object. It should use normal Contents structure.
  2. .appdev is a document package/workshop, not a bundle by default. It may contain apps, notes, research, tasks, templates, and private design process.
  3. Every Appify document package type should have an exported UTType, a document icon, package declaration, and package-local QuickLook/Preview.*.
  4. Do not put Contents/Info.plist in a document package unless the package is intentionally both a document and a bundle.
  5. Prefer Contents/SharedSupport/Appify for optional hackable examples/templates inside generated apps; prefer Contents/Resources for files required to run.
  6. Prototype .appdev Spotlight indexing before Finder Sync, because search makes appdev projects feel native without modifying Finder UI.
  7. Prototype appify: URLs and NSServices after document identity is stable, because they create command surfaces without new helper processes.
  8. Keep XPC and app extensions as escalation tools, not baseline architecture.

Open Questions

  1. Should .appdev itself ever become a true bundle with Contents/Info.plist, or should that be reserved for code-bearing objects?
  2. Should Appify UI define one canonical internal document package shape, or should each generated app own its own package vocabulary as long as Quick Look is standardized?
  3. How much legacy declaration should templates include: UTTypes only, or UTTypes plus older extension/MIME/OSType fields for older macOS targets?
  4. Would a Help Book make generated apps feel delightfully self-contained, or would it feel like ceremony compared with plain bundled docs?
  5. Is Spotlight indexing for private .appdev notes a feature or a privacy footgun?
