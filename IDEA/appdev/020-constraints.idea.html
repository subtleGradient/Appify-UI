Appify UI 26 Constraints

The idea survives if .appdev is the canonical workshop and .app is the reproducible runtime artifact. It gets fragile if the signed app is also the mutable source of truth.

Sequence: 000 capture / 010 hyperslices / 030 bundle shape / 040 two proposals

Known Facts

  • Apple documents package documents as directories that Finder and Launch Services treat as single documents when the type is registered with LSTypeIsPackage.
  • Apple also defines application bundles as application bundles. .app is already the system's strong application shape, so using .app as a document extension would fight Finder expectations and Launch Services identity.
  • AppifyHost already has three document modes: contentPackage, folderMarker, and fileDocument.
  • AppifyHost validates the package extension, rejects symlinks, derives a working directory, runs an app-local server, and accepts only loopback HTTP(S) or file URLs under the document or app bundle.
  • The current local examples already cover the needed pattern space: tw.app and litecli.app for file documents, LazyGit.app for folder markers, and TLCanvas.app for content packages.
  • A self-contained nested app cannot be the only opener for .appdev before Launch Services knows about the document type. Some installed Appify UI build has to own .appdev.
  • Opening an arbitrary .appdev is not the same as opening inert data. It may contain executable code, so the default action should inspect and validate; running nested code should be explicit.
  • Current AppifyHost Save As behavior copies whole content packages, so caches, build artifacts, node_modules, DerivedData, and logs need cleanup or exclusion policy before .appdev gets large.

Why .appdev Beats .app As Document Type

Option Golf score Reason Verdict
Native document extension is .app High badness It collides with a launchable application bundle, creates user confusion, and risks handler-ranking fights with the system application-bundle type. Keep as a fantasy target, not a first build.
Native document extension is .appdev Low badness It is a custom document package with a clear semantic boundary: the development workshop around an app. Use this.
Use .appdev as extension and "App Development Bundle" as kind Lowest naming badness Short filename, accurate Finder/document label, and one internal concept instead of split .appdesign/.appdev taxonomy. Canonical naming decision.

Failure Modes

Failure Why it matters Rescue move
Signed app vs hackable app conflict Mutating a signed/notarized bundle can break trust, quarantine behavior, update assumptions, or reproducibility. Make hacking flow through .appdev source and regeneration. The .app can be inspectable, but post-sign mutation is not the normal path.
Unclear source of truth If both .appdev and the nested .app contain canonical state, the system drifts. One rule: .appdev/appdev.json points to the target app and declares what is generated, copied, private, and shippable.
Private process leak A shipped app must not accidentally include prompts, notes, tasks, internal logs, credentials, or rough experiments. Use an allowlist packager and a leak test before every zip.
Generic host abstraction gets too big A meta-platform can become heavier than the apps it creates. Ship one path first: wrap an existing AppifyHost pattern, validate, launch, and zip.
Finder elegance becomes tool pain Bundles are directories, but many tools treat them as opaque or special. Keep plain manifests, README files, scripts, and predictable paths inside the package. Make "Show Package Contents" a feature, not a secret.
Reproducibility theater "Everything needed" can hide OS version, Xcode, Bun, signing identity, caches, secrets, fonts, or system frameworks. Define a clean-machine test and a manifest field for external prerequisites.
Document opens executable code A shared .appdev can smuggle an app, scripts, local servers, or build hooks behind a friendly package icon. Opening means inspect. Running, building, signing, and packaging are separate trust-gated actions.
Inner app update mutates trust state Replacing pieces inside a signed app can break the signature seal or create partial update states. Build Appify UI 26.app.next, verify it, then atomically swap the whole app bundle.

Tests That Should Exist Early

  • Clean machine test: copy only appify-ui.appdev to a fresh Mac user account and prove the nested app can run and be rebuilt with documented prerequisites.
  • Codesign mutation test: sign the inner app, run the intended hack workflow, and verify trust is not silently broken.
  • Source-of-truth test: ask where one UI change belongs; the answer should be obvious from the manifest and UI.
  • Leak test: package only Appify UI 26.app and search for private notes, prompts, tasks, logs, credentials, and experiments.
  • Round-trip test: regenerate the app twice from the same inputs and compare outputs.
  • Open-is-inspect test: double-click a foreign .appdev and verify no nested executable runs until the user chooses a trust-gated action.
  • Tool compatibility test: use Git, Finder, Spotlight, Zed, rg, file watchers, build scripts, and CI against the package layout.
  • Minimum app test: if the scaffolding is larger or harder to explain than the smallest generated app, the abstraction is premature.
