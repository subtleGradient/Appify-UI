Appify UI 26 Seed Idea

A new generic meta AppifyHost app called Appify UI 26.app: an app for making AppifyHost apps, with a clean split between runnable implementation and private design process.

Sequence: 010 hyperslices / 020 constraints / 030 bundle shape / 040 two proposals / 050 map reduce

Raw Capture

Let us think about a new super generic meta app called Appify UI 26.app. This will be an AppifyHost app that makes it easy and fun to make AppifyHost apps.

It would be cool if its native bundle type could be .app bundles themselves, but macOS probably will not let an app use .app as its native document bundle extension. So use .appdev instead.

Self-host the Appify UI 26 app inside an appify-ui.appdev bundle at ./appify-ui.appdev/Appify UI 26.app.

Philosophically, AppifyHost apps should be fully self-contained within their .app bundles, but only for the stuff needed to run them. The meta stuff related to the design of the app belongs in each .appdesign or .appdev bundle.

Therefore appify-ui.appdev encapsulates all the stuff related to designing Appify UI: research, specs, tasks, notes, design files, and similar material. appify-ui.appdev/Appify UI 26.app is the actual implementation.

The .app bundle should be zippable and sendable to someone else to run. It should contain all source code necessary for them to hack on it, but it should not contain private notes and personal design/dev process. They can maintain their own separate .appdev bundle with their own notes.

What The Idea Wants To Be

Fun App Forge

The target is not merely a project generator. The target is a native-feeling forge where turning a web app, TUI, or local tool into a Finder-openable Mac app feels playful and inspectable.

Clean Boundary

The runnable artifact is public and hackable. The thinking residue is private and portable between the creator's tools, but it does not leak into the distributed app.

Self-Hosting Proof

Appify UI should be made by Appify UI as early as possible. The tool earns trust by living inside the bundle structure it recommends.

Document-As-Workshop

A .appdev package is not the app. It is the workshop around the app: intent, drafts, rejected paths, assets, tests, source provenance, packaging notes, and local-only process.

Working Terms

  • .app: runnable, distributable, hackable app bundle. Contains everything needed to run and maintain the app.
  • .appdev: private or team-local App Development Bundle. Contains the workshop, research, specs, tasks, screenshots, design drafts, build recipes, and one or more app builds.
  • Appify UI 26.app: the first self-hosted implementation and test case.

Naming Decision

  • The extension is .appdev.
  • The human-readable document kind is App Development Bundle.
  • The internal manifest kind is appify.appdev.
  • .appdesign, .appwork, and .appapp stay rejected alternatives, not active sibling formats.

Open Tensions

  • Whether .appdev should be an AppifyHost document package, a workspace convention, or both.
  • How much source and toolchain belongs inside a zipped .app before the bundle becomes heavy or code-signing-hostile.
  • Whether each project needs one canonical app inside the bundle or a history of built app artifacts.
  • How private notes stay private without making the public app feel amputated or hard to modify.
  • Whether opening an .appdev should inspect the nested app by default but require explicit trust before running any nested executable.
