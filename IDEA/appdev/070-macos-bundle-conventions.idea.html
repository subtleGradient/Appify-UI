macOS Bundle Conventions To Steal

The AppifyHost move is to make app bundles and document packages feel native before inventing new meta-format machinery. macOS already has a lot of discovery surfaces hidden behind boring names: Info.plist, Contents/PlugIns, Contents/Library/Spotlight, Contents/XPCServices, .lproj, Help Books, Services, URL schemes, UTIs, and document package conventions.

Sequence: 000 capture / 030 bundle shape / 060 hello world / 080 taxonomy / Quick Look spec

Stance

Default posture

Prefer platform-discovered files and Info.plist declarations over app-specific registries when the system already has a native behavior.

Line to hold

Do not add extension/helper targets just to keep document packages aesthetically pure. Add code-bearing bundle machinery when it creates a real Finder, Spotlight, Shortcuts, sandboxing, or background-processing win.

Portability rule

Shared resources may stay platform-neutral, but a macOS package may contain macOS-specific affordance directories when those directories are the native user surface.

Pile Of Candidate Conventions

Convention Where it lives Free behavior AppifyHost use Cost shape
Package Quick Look files Document.pkg/QuickLook/Preview.*, Thumbnail.* Finder preview and thumbnail without an app extension. Default for .tlcanvas, .hi, and any readable package document. Very low. Needs save-time discipline so preview files do not drift.
Exported UTTypes UTExportedTypeDeclarations in app Info.plist System-wide type identity, conformance, kind names, icons, pasteboard/file filtering. Every AppifyHost document type should get a reverse-DNS UTType that conforms to com.apple.package for bundles. Low. Namespace care matters.
Document type declarations CFBundleDocumentTypes, LSItemContentTypes, LSTypeIsPackage Double-click open, Open With, role ranking, package treatment, document icons. Core contract for generated apps. Low. Launch Services registration can be cachey during development.
Document and app icons CFBundleIconFile, CFBundleTypeIconFile, UTTypeIconFile, Resources/*.icns Finder, Dock, Open panel, Get Info, and Launch Services visuals. Generated apps should own one app icon and each custom document type should own an icon. Low to medium. Icon production is craft work, not architecture.
Localized bundle resources Contents/Resources/*.lproj/, InfoPlist.strings, Localizable.strings Bundle resource lookup follows user language preferences. Future-proof app names, document kind names, help, and template labels without custom locale plumbing. Low if kept textual. Medium if every template becomes localized too early.
Private frameworks Contents/Frameworks/ App-local dynamic libraries that are revision-locked to the app. Could package AppifyHost shared native runtime pieces cleanly if Swift code splits. Medium. Code signing, rpaths, notarization, and binary compatibility become real.
Loadable plug-ins and app extensions Contents/PlugIns/ Blessed home for extension bundles and in-app loadable modules. Only for real extension points: Quick Look Preview extension, Finder Action extension, Share extension, Finder Sync, or app-owned plug-in APIs. Medium to high. Separate targets, signatures, entitlements, host lifecycles.
Shared support resources Contents/SharedSupport/ Named place for non-critical bundled assets such as templates, samples, tutorials, and seed data. A good home for hackable example documents or optional templates inside a self-contained app. Low. Keep runtime-required files in Resources or the app-local server tree.
Spotlight importers Contents/Library/Spotlight/*.mdimporter Searchable metadata for custom formats in Spotlight and Finder search. Tempting for .appdev, .tlcanvas, notes, task metadata, and app names embedded inside packages. Medium. Importers must be UI-free, app-independent, and privacy-aware.
XPC services Contents/XPCServices/*.xpc On-demand helper processes managed by launchd, with isolation and crash containment. Use for risky parsers, preview rendering, language servers, or build workers that should not crash the UI process. High. Protocol design, async behavior, signing, sandboxing, and statelessness matter.
Login items and bundled agents Helper executables controlled through SMAppService User-approved background agents and login helpers without loose installer scatter. Maybe for a future appdev watcher, local indexer, or app-build service. High. Background behavior is trust-sensitive and should not be casual.
URL scheme declarations CFBundleURLTypes Deep links from browsers, other apps, terminal tools, and docs. appify: links could open an appdev bundle, create an app, jump to a document, or run an app-local command. Low to medium. Global scheme ownership and security validation matter.
Services menu entries NSServices in app Info.plist System Services menu actions for selected text, files, URLs, and pasteboard data. Useful for "Make Appify app from selection", "Open package in Appify UI", or "Create note in current appdev". Medium. Old-school but native; needs pasteboard type clarity.
Finder Action extensions App extension under Contents/PlugIns Quick Actions from Finder and preview panes. Could expose "Render preview", "Validate app bundle", "Zip distributable app", or "Open in Appify UI". Medium to high. Good when file-context actions become central.
Finder Sync extensions App extension under Contents/PlugIns Badges, labels, contextual menus, toolbar buttons, and sidebar affordances for monitored folders. Maybe for a curated Appify workspace root, not for general Finder modification. High. Apple frames it around sync-like folders, not arbitrary Finder theming.
Help Books Help folder in Contents/Resources plus CFBundleHelpBookFolder and CFBundleHelpBookName Native Help menu and Help Viewer lookup. Great fit for shipped docs in hackable apps: "how this app is built" can be local and native. Low to medium. Help Book formatting is its own small tradition.
NSDocument and NSFileWrapper AppKit document architecture and package file wrappers Autosave, versions, safe package reads/writes, file coordination, recent documents. Long-term native ideal for package documents that are edited directly rather than served through shell scripts. High relative to AppifyHost's current webview host simplicity.
App Intents and App Shortcuts Swift intent types and optional extension support Shortcuts, Spotlight, Siri, and system automation discover app actions. Could surface "Create app", "Open appdev", "Export app", or "Render previews" outside the app. Medium. Best once commands are stable and user-language-friendly.
Privacy manifests Contents/Resources/PrivacyInfo.xcprivacy for macOS apps Platform-compliant privacy declaration for distribution paths that care. Part of the generated app checklist, especially if templates bundle SDKs or network code. Low if generated from known capabilities. Bad if treated as paperwork after the fact.

Adoption Ranking

Adopt by default

  • Package QuickLook/Preview.* and optional Thumbnail.*.
  • Exported UTTypes for Appify-owned document packages.
  • CFBundleDocumentTypes with LSItemContentTypes, role, icon, and package declaration.
  • App and document .icns resources.
  • Contents/SharedSupport for optional hackable examples/templates.

Prototype next

  • Spotlight importer for .appdev and .tlcanvas metadata search.
  • Help Book for generated apps that ship self-documenting source and examples.
  • CFBundleURLTypes for appify links into appdev documents and commands.
  • NSServices for selection-to-app and file-to-app workflows.
  • App Intents once the command vocabulary settles.

Defer until demanded

  • Quick Look Preview extensions, because package-local previews are cheaper and more inspectable.
  • Finder Sync, because it is for monitored-folder state, not general Finder wish-casting.
  • XPC services, unless a renderer/parser/build worker needs crash isolation or sandbox separation.
  • Login items or agents, unless background indexing/watching becomes a product-level promise.
  • Private frameworks, unless the native runtime grows past a single-app-bundle shape.

Most Promising Appify Patterns

Pattern Concrete sketch Why it feels native
Self-previewing packages Every package document has an inspectable QuickLook/Preview.* generated by the app's own save pipeline. Finder becomes a useful browser for Appify work without custom chrome.
Self-describing app bundles Generated apps include Contents/SharedSupport/Examples, a Help Book, and source in the app bundle. Zip the .app, send it, run it, inspect it, learn from it.
Searchable appdev packages A Spotlight importer extracts app name, declared document types, tasks, notes, and bundled app summaries from .appdev. App projects become first-class Finder/Spotlight knowledge objects.
Native commands without app focus appify: URLs, Services, and later App Shortcuts call stable appdev commands. The app feels like part of the Mac rather than a sealed web tool.
Crash-isolated renderers Preview rendering or untrusted package parsing moves into a bundled XPC service only when needed. AppifyHost keeps its simple host while dangerous work gets a native containment boundary.

Questions For The Next Loop

  1. Should every generated AppifyHost app reserve Contents/SharedSupport/Appify for source, examples, and templates, or should each app choose its own internal shape?
  2. Should .appdev get a Spotlight importer before Appify UI exists, or after there is enough metadata to prove the importer vocabulary?
  3. Do we want appify: URLs early as a low-cost command surface, or should we wait until command authorization and document identity are clearer?
  4. Is a Help Book charming enough to make every generated app self-documenting, or is plain bundled HTML in SharedSupport enough?
  5. Where is the line between "native Mac app" and "too much invisible OS magic" for people hacking on the generated app bundle?
