Quick Look Package Convention

Appify UI is macOS-first. Package documents should lean into macOS package conventions instead of inventing app-specific preview plumbing unless a concrete product need forces it.

Decision

  • For AppifyHost package documents, the canonical Finder preview lives under QuickLook/Preview.*.
  • When a separate thumbnail is useful, it lives at QuickLook/Thumbnail.*.
  • The QuickLook directory is not smuggled cross-platform metadata. It is the macOS package-document preview surface.
  • Do not add a Quick Look app extension merely to avoid the package-local QuickLook directory.
  • Broader ecosystem compatibility is allowed, but it is a nice-to-have layered on top of the macOS-native package contract.

Canonical Shapes

Document type Canonical preview file Notes
.tlcanvas QuickLook/Preview.png Generated from the current canvas state. QuickLook/Thumbnail.png may point at it when one PNG is sufficient.
.hi QuickLook/Preview.html The preview HTML is the document's user-visible content for the hello-world probe.

Golf Rationale

Option Thing Golf Verdict
Use QuickLook/Preview.* inside the package Low burden, high clarity, no new runtime surface. Default.
Add app-owned Quick Look extensions for every AppifyHost app Cleaner document packages, but more signing, packaging, registration, and lifecycle surface. Only when generated previews cannot be stored as native Quick Look files.
Invent app-specific preview files such as snapshot.png Creates duplicate vocabulary and adapter files. Avoid when the file exists primarily for Finder/user preview.
