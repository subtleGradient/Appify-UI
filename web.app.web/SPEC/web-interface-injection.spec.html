Web Interface Injection

Web.app should let .web bundles depend on stable interfaces instead of specific component implementations. The first version is static, manifest-declared, and import-map friendly: it chooses web-native files, not npm packages, install scripts, or framework adapters.

This builds on Stable Webspace Origins and Native Web Compatibility: peer bundles already share a same-origin URL space, so the missing piece is a small, inspectable matching convention.

Decision

  • Use manifest-declared interfaces and providers for the first dependency-injection surface.
  • Resolve interface dependencies into a normal browser import map before page modules run.
  • Keep interface matching separate from npm, JSR, TypeScript compilation, lifecycle scripts, and build steps.
  • Define JSON Schema documents for static descriptor validation; allow Standard Schema as optional executable conformance metadata later.
  • Model the built-in matchmaker as a virtual /_web/matchmaker.web/ bundle that is also mirrored as static public web files.
  • Support multiple providers for one interface so a consumer can select by theme, behavior, provenance, or local availability.
  • Fail loudly when no provider satisfies a required interface; never silently fetch or install a replacement.

Virtual Matchmaker Bundle

  • /_web/matchmaker.web/ is a built-in static bundle served by Web.app, not a hidden native page global.
  • The same files should be publishable at a public URL so normal static hosting can validate and resolve the same descriptor shapes.
  • The matchmaker reads consumer requirements and provider descriptors that are already visible web files.
  • The primary output is a concrete import map plus optional diagnostics, not a bespoke module loader.
  • The public mirror may recommend CDN URLs, but local Web.app resolution should prefer same-webspace providers when they exist.

Resolution Rules

  1. Read the consumer's declared import requirements.
  2. Find provider descriptors in same-webspace .web peers, then explicit remote catalogs if the consumer names them.
  3. Reject providers whose interface id or version is incompatible.
  4. Rank compatible providers by explicit consumer preference, same-webspace locality, pinned provenance, and deterministic name order.
  5. Emit an import-map prefix for each logical specifier.
  6. Show a human-readable failure page or diagnostic if any required interface is unresolved.

Import-map generation may be a host feature later. The first fixture can hand-write the resolved import maps to prove the interface boundary before adding host machinery.

Security And Portability

  • No package install happens when a .web opens.
  • No lifecycle script runs as part of dependency resolution.
  • No moving remote dependency is required for a local package to open once its providers are present.
  • Remote providers should be commit-pinned Git sources, content-addressed materializations, or explicit static URLs.
  • Provider descriptors are promises about shape, not proof of trust. Provenance and signatures can be layered later.
  • Consumers should depend on custom element names, DOM attributes, events, CSS variables, and ESM exports that work in normal browsers.

Acceptance Tests

  • A consumer imports @web/ui-controls/define.js without knowing which provider implements it.
  • Two provider entrypoints render the same consumer app with different visual behavior or theme.
  • Both providers dispatch the same standardized DOM events with compatible detail shapes.
  • Switching providers changes only the import map and provider stylesheet.
  • A missing provider fails visibly instead of reaching for npm or a default CDN.
  • The same descriptor files can be served as ordinary static web files outside Web.app.

Golf Rationale

Option Liability Verdict
Descriptor-to-import-map matching Small new convention, keeps browser-native ESM and static hosting. Default.
Runtime broker API More dynamic, but teaches pages to depend on a Web.app-only service. Defer.
Service-worker module rewriting Powerful but harder to inspect, cache, and debug. Only if static maps cannot carry real examples.
npm-style registry Imports install-time scripts, account incentives, and supply-chain pressure. Reject for .web.
TypeScript-first package identity Good developer tooling, but couples static web packages to a compile-time ecosystem. Use as optional authoring aid only.
