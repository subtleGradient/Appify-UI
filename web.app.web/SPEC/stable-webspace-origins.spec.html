web.app.web / SPEC / origins and storage

Stable Webspace Origins

Web.app serves .web bundles as normal client-side web sites. AppifyHost should give each webspace a stable localhost origin so native browser persistence, service workers, and cross-bundle composition behave like they would on a normal hosted HTTPS site.

See also Native Web Compatibility, which defines how Web.app should support Web 1.0, early AJAX, Rails-era, and PWA-style conventions through standard client-side web APIs instead of Appify-specific request globals.

Decision

  • Use one fixed AppifyHost-visible webspace port: 55555.
  • Derive the visible hostname from the webspace root, not from the opened .web bundle.
  • Load Web.app windows from stable origins shaped like http://repo-slug--rootHash.localhost:55555/path/to/a.web/.
  • Keep the actual Runner/Bun server port as a private backend implementation detail.
  • Never require the Runner/Bun process to bind TCP port 55555 for normal operation.
  • Native website-specific storage should persist normally through WebKit's persistent website data store.
  • File-backed localStorage remains the intentional Web.app document-storage enhancement.

Visible Origin vs Transport Endpoint

This is the invariant that should prevent future confusion: WebKit sees the stable origin; the backend uses an ephemeral private port.

Layer Who Sees It Example Contract
Visible webspace URL WebKit, page JavaScript, service workers, native website storage. http://appify-ui--6e3025a4.localhost:55555/ideas.web/ Stable across restarts; this is the origin used for IndexedDB, CacheStorage, cookies, service workers, OPFS, and native browser policy.
Hidden backend URL AppifyHost and the local transport layer. http://127.0.0.1:49152/ideas.web/ May change on every launch; this is only where the Bun Runner listens.
  • 55555 is part of the browser origin contract, not the Bun listen-port contract.
  • AppifyHost should map visible requests to the hidden backend, preferably with WKWebsiteDataStore.proxyConfigurations on macOS 14 and newer.
  • Directly binding 55555 is not part of the design; Web.app must not depend on a real TCP listener on that port.
  • A port conflict on 55555 should not normally stop Web.app from opening, because the backend should not need that port.

Reasoning DAG

Read top to bottom: values select which constraints matter, constraints narrow the choices, decisions produce the implementation contract, and outcomes become acceptance criteria.

  • Native web plus origin rules imply a stable visible origin mapped to a hidden dynamic backend port.
  • Composition plus existing repo path geometry imply one hostname per webspace root instead of one hostname per bundle.
  • Pit-of-success security plus filesystem writes imply strict file-key routing with ordinary-storage fallback.
  • Low bookkeeping plus localhost trust imply visible port 55555, not remembered per-bundle visible ports or cert plumbing.

Origin Model

Concept Identity Example

LocalStorage Storage Semantics

  • Ordinary localStorage keys are shared across the webspace and persisted in <webspace-root>/.local/storage.json5.
  • ./file.ext keys write files relative to the current page URL directory.
  • /file.ext keys write files relative to the shared webspace root.
  • Invalid or unsupported file keys are not errors; they fall back to the shared .local/storage.json5.
  • Bundles that want private non-file state should namespace normal keys with their route path.

Examples

// From /path/to/a.web/editor.html:
localStorage.setItem("theme", "dark")
// => repo/.local/storage.json5

localStorage.setItem("./document.canvas", stableJSON(canvas))
// => repo/path/to/a.web/document.canvas

localStorage.setItem("/path/to/sub/b.web/document.canvas", stableJSON(canvas))
// => repo/path/to/sub/b.web/document.canvas

const scope = new URL(".", location.href).pathname
localStorage.setItem(`${scope}:view`, JSON.stringify(view))
// => repo/.local/storage.json5, namespaced by bundle path

File-Key Security

  • A separate file is created only when the key starts with / or ./.
  • The key must pass strict path grammar: no .., percent escapes, backslashes, null bytes, empty segments, query strings, fragments, hidden segments, or control directories.
  • The final segment must have a file extension.
  • The resolved route must land inside a known writable web root.
  • Known writable roots include the active local .web package root and local peer manifest source roots.
  • Remote and materialized git cache mounts are read-only for file-backed localStorage writes.
  • Existing directory chains must not traverse symlinks.

Shared Disk Index

The root storage file must merge writes from all bundles in the webspace. File entries are indexed by resolved route path so multiple bundles can use the same relative key, such as ./document.canvas, without clobbering each other. 

{
  schema: 4,
  entries: [
    { key: "theme", value: "dark" },
    { key: "/path/to/a.web/:view", json: { selectedId: "intro" } },
  ],
  files: [
    {
      key: "./document.canvas",
      routePath: "/path/to/a.web/document.canvas",
      valueType: "text",
    },
    {
      key: "./document.canvas",
      routePath: "/path/to/sub/b.web/document.canvas",
      valueType: "text",
    },
  ],
}
  • Hydration includes all ordinary shared entries.
  • Hydration includes explicit /... file entries because their root-relative route is unambiguous.
  • Hydration includes ./... file entries only when they resolve for the current page path.
  • Writes must merge with the existing file index instead of replacing unrelated sibling-bundle file entries.

Cross-Bundle Composition

  • A repo webspace shares one origin across all served .web bundles under that repo.
  • Imports and fetches between peer bundles stay same-origin and do not require CORS.
  • Service worker scopes follow normal web path rules.
  • Opening a peer bundle as its own document window keeps the same webspace origin if it belongs to the same repo root.
  • The path, not the hostname, distinguishes bundle identity inside a composed webspace.

Implementation Notes

  • AppifyHost owns the visible *.localhost:55555 origin and maps stable webspace hostnames to hidden dynamic backend URLs.
  • The Bun Runner should bind an ephemeral loopback port by default; 55555 is the WebKit origin port, not the Runner listen port.
  • If WebKit routing requires an HTTP CONNECT proxy, the Runner may bind a second ephemeral loopback tunnel that only accepts the exact stable webspace host and forwards to the backend.
  • Prefer names such as visibleWebspaceURL, backendServerURL, and routeVisibleOriginThroughBackend so code does not blur the two layers.
  • Hostnames are deterministic: <safe-webspace-name>--<short-root-hash>.localhost.
  • Do not introduce a real listener on 55555, even as a migration shortcut.
  • Web.app should use a persistent WebKit website data store for native site-specific persistence.
  • /_web/* write and control routes should require same-origin checks plus an unguessable per-document control token.
  • Do not patch normal web APIs such as fetch, Workers, IndexedDB, CacheStorage, cookies, OPFS, service workers, dialogs, or window.open merely to police native browser behavior.

Acceptance Tests

  • window.location.origin stays stable across backend restarts.
  • The visible URL uses port 55555 even when the backend Runner listens on an ephemeral loopback port.
  • window.isSecureContext is true on the stable *.localhost:55555 origin.
  • IndexedDB, CacheStorage, cookies, service workers, and OPFS persist per webspace origin after reopening Web.app.
  • Two different repo roots get isolated origins and cannot read each other's native website storage.
  • Peer bundles in the same webspace can import and fetch each other's files without CORS.
  • Ordinary localStorage keys from multiple bundles merge into one root .local/storage.json5.
  • ./document.canvas from two different bundle paths writes two different files and keeps two distinct file index entries.
  • Unsafe file keys fall back to root .local/storage.json5.
  • Remote and git-cache mounts cannot be written through file-backed localStorage keys.

Golf Rationale

Option Thing Golf Verdict
Webspace-root origin with shared fallback storage Matches normal hosted website semantics, minimizes CORS, and keeps file-backed writes behind strict route checks. Default.
One origin per opened bundle Improves app isolation but makes cross-bundle composition feel less like a normal site. Avoid for repo webspaces.
Dedicated remembered ports per bundle Adds bookkeeping, collisions, migrations, and stale state. Reject.
Portless HTTPS localhost Cleanest URL, but requires certs, privileged ports, or helper plumbing. Defer until fixed-port localhost proves insufficient.
